# 004 — Discord auto-unlock — History

## 2026-07-14 18:39 +07 — Initial implementation

**Context:** User hits medoctruyen.vn's hard 50-chapters/day cap while downloading;
the site issues a single-use code and requires typing `/mochuong <code>` in its
Discord `#mở-khoá` channel to unlock the next 50. Previously manual; user wants it
automated.

**Decision (with user):** Chose **Option A — browser automation on a throwaway
Discord account**. Rejected: token self-bot (higher ban risk) and OS keystroke
injection (fragile, needs Accessibility perms). Kept a one-click manual *assist* as
the fallback path. Playwright drives a dedicated, persistent Chromium profile
(separate from the user's normal browser); `/mochuong` is a real slash command so we
type name → wait for picker → Enter → type code → Enter.

**Changes:**
- `discord_unlock.py` (new): lazy-Playwright automation — `open_login()`,
  `run_unlock()`, `valid_channel_url()`, dedicated `profile_dir()`
  (`<library>/.discord-profile`).
- `errors.py` + `scrapers/medoctruyen.py`: `DailyLimitError` now carries structured
  `code` + `discord_url`; scraper refactored `_daily_limit_message` →
  `_daily_limit_details` returning `(message, code, discord_url)`.
- `gui/workers.py`: `UnlockWorker` + `DiscordLoginWorker` (off-thread, Playwright
  blocks); `daily_limit_hit` signal now `(message, code)`.
- `gui/tab_scrape.py`: auto-unlock + auto-resume wiring; runaway-loop guard
  (`_MAX_AUTO_UNLOCKS = 3` unlocks with no chapter progress → fall back to manual);
  manual fallback dialog copies `/mochuong <code>` to clipboard + "open channel".
- `gui/settings_dialog.py`: "Tự mở khoá qua Discord" section — enable checkbox,
  channel-URL field, "Đăng nhập Discord" button, throwaway-account/ToS hint;
  validates channel URL on save.
- `pyproject.toml`: new optional `discord` group (`playwright>=1.40`).
- Tests: `test_discord_unlock.py` (channel-URL validation, run_unlock guards,
  profile dir); extended medoctruyen daily-limit test to assert `code`/`discord_url`.

**Verification:** ruff clean; 145 passed (test_tts.py skipped — pre-existing missing
`numpy`/tts optional dep); core + GUI modules import without Playwright installed;
lazy-import + input guards confirmed.

**Open items / caveats:**
- Discord web selectors (composer, command picker) were NOT tested against a live
  Discord session — written defensively; may need tuning on first real run
  (`headless=False` in `run_unlock` to watch).
- ToS: automating a Discord user account violates Discord ToS regardless of method —
  throwaway-account guidance + once/day cadence.
- Not committed; left for the user's `/ship` flow. `uv.lock` remains untracked
  (pre-existing, unrelated).

## 2026-07-14 — Live validation + hardening (send-verification, real Chrome, resume race)

**Context:** Reviewed the committed feature, then drove it against a live Discord
session. Fixed the issues that review and live testing surfaced.

**Review fixes (before live test):**
- `run_unlock` no longer reports success just because it didn't throw. It now asserts
  the command picker actually offers `/mochuong` before pressing Enter (`_command_offered`)
  and that the composer empties after send (`_wait_until_sent`) — otherwise it raises,
  so the caller never resumes on a command that silently went nowhere or, worse, posted
  `/mochuong`+code as two plain-text messages. Both failure paths `_clear_composer` to
  avoid poisoning the next run's draft.
- Manual fallback (`_on_unlock_failed`) now passes the real `_last_limit_code` through
  (was `""`), so the clipboard pre-copy + open-channel button work on that path too.

**Login wouldn't work in Playwright's bundled Chromium** (captcha/device-check refused
it). `_launch_context` now launches the *installed* Chrome (`channel="chrome"`, bundled
Chromium as fallback), drops `--enable-automation` + sets `--disable-blink-features=
AutomationControlled` (navigator.webdriver → false). Login then succeeded (QR + captcha,
user-driven). `run_unlock` default flipped to `headless=False` — headless UA invites a
mid-run re-challenge on this hard-gated profile.

**Live dry-run (real channel, stops before send):** selectors confirmed against
`channels/1512873734707351746/1516869015463792741`. Composer matched; picker offered
the real `/mochuong [code]` app command ("Unblock Bot"); command armed as a proper
slash-command pill with the code in the argument. Channel history showed other users'
plain-text `/mochuong CODE` mistakes — exactly what the new `_command_offered` guard
prevents.

**Resume-race fix (user-reported "unlocks twice"):** "command sent" ≠ "cap lifted" —
the bot needs a moment. `_on_unlocked` resumed instantly, re-hit the limit page (fresh
single-use code), and fired a redundant second unlock. Added `_UNLOCK_SETTLE_MS = 6000`
grace via a single-shot `_resume_timer` (owned by the tab, stopped in `shutdown()`);
`_auto_unlocking` now cleared in `_start_download` instead of `_on_unlocked` so a
normally-completing resume still shows its finished UI.

**Config set:** auto-unlock enabled + `#mở-khoá` channel URL saved.

**Tests:** +`TestLaunchContext` (Chrome-first/fallback/no-browser), `_command_offered`,
`_wait_until_sent`. 180 passed, ruff clean.

**Still unverified:** the final send + `_wait_until_sent` on a *real* code (needs an
actual cap-hit); `_UNLOCK_SETTLE_MS = 6000` is a first guess at bot latency (the
`_MAX_AUTO_UNLOCKS = 3` guard covers an under-estimate). ToS caveat unchanged.
