# NovelTrans ŌĆö Implementation Plan
## Python Desktop GUI: Scrape ŌåÆ Translate ŌåÆ Export Chinese Web Novels

## 1. Overview

NovelTrans is a PySide6 desktop app with 3 tabs: (1) scrape/download a novel from a Chinese novel website, (2) translate chapters from Chinese to Vietnamese or English (free Google Translate or Claude API), (3) export the translated novel to DOCX, Markdown, or EPUB. The core design is a plugin architecture: `SiteAdapter` for scrapers, `Translator` for translation engines, `Exporter` for output formats ŌĆö all plain-Python and GUI-independent, so the whole pipeline works and is testable without Qt. Each novel is stored as an inspectable "project folder" (JSON metadata + SQLite chapter store) under a user-configurable library directory.

### 2.2 Core data classes (`models.py`)

```python
@dataclass
class NovelMeta:
    url: str
    site: str                 # adapter name, e.g. "ixdzs"
    title: str
    author: str = ""
    description: str = ""
    cover_url: str = ""

@dataclass
class ChapterRef:              # from the table-of-contents scan
    index: int                 # 0-based order
    title: str
    url: str

@dataclass
class Chapter:
    index: int
    title: str
    url: str
    content: str = ""          # original Chinese text ("" = not downloaded)
    translated: str = ""       # "" = not translated
    translated_title: str = ""

    error: str = ""
```

### 2.3 Key abstractions (real signatures)

**`scrapers/base.py`**

```python
class SiteAdapter(ABC):
    name: str                          # "ixdzs"
    display_name: str                  # "ńł▒õĖŗńöĄÕŁÉõ╣” (ixdzs8.com)"
    url_patterns: list[str]            # regexes to match novel URLs

    def __init__(self, client: HttpClient): ...

    @classmethod
    def matches(cls, url: str) -> bool: ...

    @abstractmethod
    def fetch_metadata(self, url: str) -> NovelMeta: ...

    @abstractmethod
    def fetch_chapter_list(self, url: str) -> list[ChapterRef]: ...

    @abstractmethod
    def fetch_chapter(self, ref: ChapterRef) -> str:
        """Return plain-text chapter content. Raise ObfuscatedContentError
        if the site returns content that cannot be decoded."""
```

`HttpClient` wraps `requests.Session` with: browser-like User-Agent, configurable `delay_seconds` (sleeps between requests), retry with backoff (3 tries), and automatic encoding fix (many CN sites are gbk). Registry in `scrapers/__init__.py`: `adapter_for_url(url, client) -> SiteAdapter | None`.

**`translators/base.py`**

```python
class Translator(ABC):
    name: str                          # "google" | "claude"

    @abstractmethod
    def translate(self, text: str, source: str = "zh",
                  target: str = "vi") -> str: ...

    def translate_chapter(self, title: str, content: str,
                          source: str = "zh", target: str = "vi"
                          ) -> tuple[str, str]:
        """Default: chunk content (google: ~4500 chars on paragraph
        boundaries; claude: whole chapter per request), translate,
        rejoin. Returns (translated_title, translated_content)."""
```

- `google_free.py`: `deep_translator.GoogleTranslator` ŌĆö free, no key, must chunk under ~5000 chars, throttled by the same delay setting.
- `claude.py`: Anthropic SDK, model configurable (default a cheap fast model, e.g. Haiku-class), system prompt: "Translate this Chinese web-novel chapter to {Vietnamese|English}. Keep paragraph breaks, translate names consistently, output only the translation."

**`exporters/base.py`**

```python
class Exporter(ABC):
    name: str                  # "docx" | "markdown" | "epub"
    extension: str             # ".docx" ...

    @abstractmethod
    def export(self, meta: NovelMeta, chapters: list[Chapter],
               out_path: Path, use_translation: bool = True) -> Path: ...
```

Each exporter writes a title page/front matter (title, author, description, "Translated by NovelTrans") then one section per chapter, skipping chapters with empty text (and listing skipped ones in front matter).

### 2.4 Threading model (Qt)

- All long jobs run in `QThread` subclass workers in `gui/workers.py` ŌĆö never on the GUI thread.
- Workers hold no Qt widgets; they call the plain-Python core and communicate only via signals:

```python
class DownloadWorker(QThread):
    progress = Signal(int, int, str)   # done, total, chapter title
    chapter_done = Signal(int)         # chapter index (table row refresh)
    error = Signal(int, str)           # chapter index, message
    finished_ok = Signal()

    def __init__(self, project: NovelProject, adapter: SiteAdapter): ...
    def cancel(self): self._cancelled = True   # checked between chapters
```

Same shape for `ScanWorker` (metadata + TOC), `TranslateWorker`, `ExportWorker`. One worker of each kind at a time (buttons disabled while running); Cancel is cooperative between chapters, so a chapter is never half-written. Workers write to `NovelProject` directly (single writer at a time ŌåÆ no SQLite locking issues); the GUI re-reads rows on `chapter_done`.

### 2.5 Data flow

```
URL ŌåÆ adapter_for_url() ŌåÆ ScanWorker ŌåÆ NovelProject created (meta + chapter rows "pending")
    ŌåÆ DownloadWorker ŌåÆ fetch_chapter() per row ŌåÆ content saved, status="downloaded"
    ŌåÆ TranslateWorker ŌåÆ translate_chapter() per row ŌåÆ translated saved, status="translated"
    ŌåÆ ExportWorker ŌåÆ Exporter.export() ŌåÆ .docx/.md/.epub file
```

Tabs 2 and 3 have a novel-picker (combo box fed by `Library.list_projects()`), so the app can resume any project after restart.

## 3. Data model & storage

**Decision: project folder per novel = `meta.json` + SQLite for chapters.** Rationale: SQLite gives atomic resumable status updates and fast queries over thousands of chapters (one file, no 3000-tiny-files folder); `meta.json` keeps the project human-inspectable; an explicit "Export originals/translations as .txt files" is trivial to add later. Pure per-chapter JSON files were rejected because status bookkeeping and partial-write safety get messy at 2000+ chapters.

```
<library_dir>/                       # default ~/NovelTrans, changeable in Settings
ŌööŌöĆŌöĆ <slug>-<hash8>/                  # e.g. "zhanshen-7328655521/"
    Ōö£ŌöĆŌöĆ meta.json                    # NovelMeta + created_at + counts
    Ōö£ŌöĆŌöĆ chapters.db                  # SQLite
    Ōö£ŌöĆŌöĆ cover.jpg                    # optional
    ŌööŌöĆŌöĆ exports/                     # default output dir for Tab 3
```

`chapters.db` schema (created by `storage/project.py`, `PRAGMA journal_mode=WAL`):

```sql
CREATE TABLE chapters (
  idx              INTEGER PRIMARY KEY,   -- chapter order
  title            TEXT NOT NULL,
  url              TEXT NOT NULL,
  content          TEXT NOT NULL DEFAULT '',
  translated       TEXT NOT NULL DEFAULT '',
  translated_title TEXT NOT NULL DEFAULT '',
  target_lang      TEXT NOT NULL DEFAULT '',   -- lang of `translated`
  status           TEXT NOT NULL DEFAULT 'pending',
  error            TEXT NOT NULL DEFAULT '',
  updated_at       TEXT NOT NULL DEFAULT ''
);
```

`NovelProject` API: `create(library_dir, meta, refs)`, `open(path)`, `chapters() -> list[Chapter]`, `pending_download() / pending_translation(target_lang)`, `save_content(idx, text)`, `save_translation(idx, title, text, lang)`, `mark_error(idx, msg)`, `counts() -> dict`. Resume = query `pending_*` and skip the rest ŌĆö works for both download and translation for free. If the user changes target language, rows with a different `target_lang` count as pending again (old translation overwritten after confirmation dialog).

**Settings** (`config.py`): `QSettings("noveltrans", "noveltrans")` ŌĆö keys: `library_dir`, `request_delay` (default 1.5s), `target_lang` (vi|en), `translator` (google|claude), `claude_api_key`, `claude_model`. Settings dialog reachable from a toolbar/menu on `MainWindow`. (QSettings chosen over config.toml: zero file management, native per-OS location; note plainly in README that the API key is stored unencrypted.)

## 5. Key risks & mitigations

1. **fanqie obfuscation (biggest risk).** fanqienovel.com deliberately scrambles chapter text: web pages render some characters via a custom font whose glyphs are remapped (the HTML codepoints ŌēĀ real characters), and the mobile/API path uses encrypted endpoints. Community projects (e.g. the `fanqie-novel-download` family on GitHub) handle this by (a) maintaining a codepointŌåÆcharacter mapping table for the custom font, and/or (b) calling third-party/reverse-engineered API endpoints that return clean text ŌĆö those endpoints break regularly. *Mitigation:* ship the glyph-mapping approach with the table stored as a data file (`scrapers/fanqie_charmap.json`) so it's updatable without code changes; treat any unmapped codepoint above a small threshold as decode failure ŌåÆ `ObfuscatedContentError` ŌåÆ per-chapter error status + honest UI message, never silent garbage text. Do NOT build the app's success criteria on fanqie: milestones M2-M7 guarantee the product works on cleaner sites regardless.
2. **Rate limiting / IP blocking.** All requests go through one `HttpClient` with a configurable delay (default 1.5s), realistic User-Agent, retry-with-backoff, and sequential (not parallel) chapter fetching. Resume support means a mid-run block loses nothing.
3. **Site markup drift.** Mirror sites change domains/markup often. Selectors live in class-level constants; offline HTML fixtures make breakage obvious in tests; adapters raise `ScrapeError` with the URL for easy diagnosis.
4. **Google free translate throttling/quality.** deep-translator's free endpoint can throttle or drift. Chunk Ōēż4500 chars, reuse the request delay, retry x3, then mark chapter `error` (retryable). Claude engine is the quality/reliability alternative.
5. **Claude cost/limits.** Chapter-per-request with a cheap default model; show estimated chapter count before starting; per-chapter persistence means an interrupted run only re-pays for unfinished chapters. Handle 429/overloaded with backoff.
6. **Packaging.** PySide6 + PyInstaller works but is fiddly (Qt plugins). Keep it out of scope until M9; primary distribution is `pip install -e .` + `noveltrans`.

## 7. Testing strategy

- **Adapters:** offline-first ŌĆö one saved HTML fixture per page type (novel page, TOC, chapter) in `tests/fixtures/<site>/`; tests parse fixtures and assert exact metadata/chapter-count/content snippets. `responses` maps fixture URLs to files so `HttpClient` code paths (encoding, retry) are exercised too. A separate `@pytest.mark.live` suite (excluded from default run) hits real sites for drift detection.
- **Translators:** mock `deep_translator.GoogleTranslator.translate` and `anthropic.Anthropic.messages.create`; assert chunking on paragraph boundaries, rejoin order, retry/backoff, and that chunk boundaries never split a paragraph.
- **Exporters:** golden-file for Markdown; for DOCX/EPUB, re-open output with python-docx/ebooklib and assert title, chapter headings, chapter count, and skipped-chapter note.
- **Storage:** temp-dir project lifecycle: create ŌåÆ save ŌåÆ reopen ŌåÆ resume queries ŌåÆ error/retry transitions; target-language change re-pends rows.
- **GUI:** mostly manual, using milestone "runnable checks" as the test script. Workers are testable headlessly (they depend only on core objects); a couple of `pytest-qt` signal-emission tests are optional, not a milestone gate.
- CI-friendly: everything except `@pytest.mark.live` runs offline; `pytest` in CI needs no display server since GUI tests are manual/optional.
