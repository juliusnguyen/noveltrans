# 024 — YouTube Auto-Upload (batch) — Implementation Plan

## Summary / approach

After a batch video render, upload each produced part-video to YouTube with an auto-generated title, a structured description (original + Vietnamese title/author, whole-novel chapter count, a per-part chapter-timestamp table), and an auto-generated thumbnail composited from a user-chosen base image.

The work splits cleanly into **pure builders** (title/description strings, thumbnail composition math) that are unit-testable with no network or ffmpeg, and **I/O layers** (OAuth, resumable upload, a QThread worker) that mirror the existing `VideoWorker` conventions. To keep timestamps accurate and decouple upload from render, `render_video` will emit a small **sidecar JSON manifest** next to each `.mp4` capturing the part number, chapter boundaries, and real per-chapter durations; the upload worker consumes that manifest, so it never re-probes audio or re-reads the DB.

Upload is a **separate, explicit step** (opt-in checkbox to auto-chain after render, plus a standalone "Tải lên YouTube" button for already-rendered parts), because uploads are quota-expensive (~6/day) and must be resumable/idempotent — a batch that dies partway must not re-upload parts that already succeeded.

New code lives in a new `src/noveltrans/youtube/` package (metadata builders, thumbnail, oauth, upload client) plus a new `YouTubeUploadWorker` in `gui/workers.py` and GUI wiring in `gui/tab_audio.py`.

## Decisions on the open questions

**1. OAuth flow & credential storage.**
Use the **installed-app loopback (127.0.0.1) flow**, hand-rolled over `requests` + Python's `http.server` — no `google-api-python-client`/`google-auth-oauthlib` dependency. Rationale: the app is already `requests`-based and PyInstaller-frozen; the Google client libraries pull in a large transitive tree (`httplib2`, `google-auth`, `protobuf`, `uritemplate`, `pyparsing`) that adds bundle weight, hidden-import surface, and code-signing risk for a flow that is ~120 lines by hand. The loopback flow is preferred over the device flow because the desktop app can open a browser and bind a localhost port; device flow is for input-constrained devices and gives a worse UX here.

- **Client secret (OAuth client_id/client_secret for a "Desktop app" client):** the **user must supply their own** Google Cloud project + OAuth client. Rationale: a bundled client secret in a distributed desktop app is not truly secret and a single shared client shares the 10k-unit/day quota across all users (≈6 uploads/day total). Each user gets their own quota by using their own project. Store `youtube_client_id` / `youtube_client_secret` in `QSettings` via new `AppConfig` properties (matching the existing plaintext-secret pattern for the Claude key). Provide clear Vietnamese setup instructions in the UI/tooltip.
- **Token storage:** persist the OAuth **refresh token + access token** as a JSON blob in a per-user file under the app config area, not scattered across QSettings keys. Recommend `AppConfig.library_dir / ".youtube-token.json"` (or a dedicated `youtube_token` QSettings string holding the JSON). Refresh the access token on demand using the stored refresh token; only trigger the interactive browser flow when there is no valid refresh token.
- **Scope:** `https://www.googleapis.com/auth/youtube.upload` (upload + thumbnail set). Reading playlists (optional feature) would additionally need `youtube.readonly` or `youtube` — keep playlist assignment behind the broader scope only if enabled.

**2. Where novel metadata is sourced (and gaps).**
- Vietnamese title: `project.meta.translated_title` (fallback `title`).
- Original Chinese title: `project.meta.title`.
- Original author: `project.meta.author`.
- **Vietnamese author — GAP: `NovelMeta` has no `translated_author`.** Add a `translated_author: str = ""` field to `NovelMeta` (safe: `from_dict` filters unknown keys, so old `meta.json` files load fine; `to_dict` writes it going forward). Persist it via an extended `save_meta_translation(...)` (add an optional `author` param). Populate it wherever `translated_title`/`translated_description` are produced (the meta-translation step). **Fallback when empty:** omit the Vietnamese author quoted segment (render `Tác giả: {author}` only) rather than emit an empty `""`.
- Total chapter count (whole novel): `project.counts()["total"]`.
- Per-chapter titles: `chapter_marker_title(c)` (already used to build segments).

**3. Part boundaries + per-chapter timestamps at render time.**
`plan_merge_windows(...)` is the source of truth; **part number N = window index + 1** (`i` in the `VideoWorker` loop). Timestamps come from `_with_real_durations(segments)` (ffprobe), already computed inside `render_video`. To carry this to the (later, separate) upload step accurately, have `render_video` write a **sidecar manifest** `<out>.mp4.json` containing: `part_number`, `first_num`, `last_num`, and an ordered list of `{title, seconds}` using the real probed durations. `VideoWorker` passes `part_number=i+1` into `render_video`. The upload worker reads the manifest — no DB round-trip, no re-probe, and the chapter table exactly matches the rendered audio.

**4. Privacy & playlist.**
Default **privacy = `unlisted`**, configurable in the UI (`public` / `unlisted` / `private`) via a new `AppConfig.youtube_privacy`. Scheduled publishing (`publishAt`) is out of scope for v1 (note as a follow-up; it requires `private` + an RFC3339 timestamp). **Playlist = optional**, off by default: a `youtube_playlist_id` config string; when set, after each successful upload call `playlistItems.insert` (cheap, ~50 units). Keep playlist behind a checkbox so the default upload scope stays `youtube.upload`-only.

**5. Thumbnail rendering.**
Use **Pillow** (`PIL.ImageDraw` + `ImageFont`), not the ffmpeg preview renderer. Rationale: the thumbnail is a single full-bleed still with multi-line glow text and a large centered `PHẦN {N}` — this is exactly what Pillow does well (the codebase already composits with Pillow in `player_skin.py`: gradients, masks, `GaussianBlur` shadows). The ffmpeg music-player skin has the wrong layout (photo framed left, vinyl, bars) and is animation-oriented. Reuse the bundled fonts and the `font_dir_context()` accessor.

- **Decorative glow font:** the example uses a script/calligraphic glow face that is **not bundled** and most script fonts lack full Vietnamese diacritic coverage. Decision for v1: **approximate with a bundled font** (`PlayfairDisplay-Regular.ttf`, the most decorative of the six, full Vietnamese coverage) **plus a Pillow "glow" effect** — render the text onto a transparent layer, duplicate it blurred in a cyan/white tint (`GaussianBlur`) as an outer glow, then draw a dark stroke + light fill on top. This matches the example's glow look without a new asset. Follow-up option: bundle one true display face with verified Vietnamese coverage (e.g. a Google Fonts OFL calligraphic family that covers `latin-ext` + Vietnamese) and add it to `VIDEO_FONTS`.

## Phase 1 — Metadata: `translated_author` + pure title/description builders

**Modify `src/noveltrans/models.py`:** add `translated_author: str = ""` to `NovelMeta` (after `translated_description`). No migration code needed (dataclass + `from_dict` key-filtering handle old files).

**Modify `src/noveltrans/storage/project.py`:** extend `save_meta_translation(self, title, description, lang, author="")` to also write `translated_author` into `self.meta` and `meta.json`. (Find current callers of `save_meta_translation` and pass author through where a translated author is available; otherwise leave `""`.)

**New `src/noveltrans/youtube/metadata.py`** (pure, no I/O):
- `build_upload_title(translated_title: str, part_number: int) -> str` → `"{translated_title} - Phần {N}"`.
- `build_chapter_timestamps(segments) -> list[tuple[str, str]]` — reuses `_yt_timestamp` logic; returns `[(timestamp, title), ...]`, first always `0:00`. (Factor `_yt_timestamp` out of `tts/video.py` into a shared spot or import it — keep one implementation.)
- `build_upload_description(*, original_title, translated_title, original_author, translated_author, total_chapters, segments, credit="Fox Novel") -> str` — emits the exact shape from the prompt:
  ```
  Tên truyện: {original_title} — "{translated_title}"
  Tác giả: {original_author} "{translated_author}"   # author line omits the quoted VN part when translated_author == ""
  Số chương: {total_chapters}

  Mục lục chương:
  0:00 {ch1}
  5:51 {ch2}
  ...

  Tạo bởi: {credit}
  ```
- Both functions are import-free of Qt/ffmpeg → directly unit-testable.

The manifest can drive these builders: a small `PartManifest` dataclass (`part_number`, `first_num`, `last_num`, `chapters: list[{title, seconds}]`) with `to_dict`/`from_dict`.

**Note on the existing `build_youtube_description(segments, novel_title)`** in `tts/video.py`: keep it (it still writes the human-friendly `<out>.txt` companion), but consider having it delegate its timestamp table to `build_chapter_timestamps` to avoid two timestamp implementations.

## Phase 2 — Sidecar manifest at render time

**Modify `src/noveltrans/tts/video.py` (`render_video`):** add a `part_number: int | None = None` parameter; after the render, write `<out>.mp4.json` (a `PartManifest`) using the already-probed `segments` (post `_with_real_durations`). This is a tiny pure serialization; guard so it's skipped when `part_number is None` (backward compatible).

**Modify `src/noveltrans/gui/workers.py` (`VideoWorker.run`):** pass `part_number=i+1` into `render_video`. (Batch mode: window index +1 is the Part ordinal; for `all`/`range` single-file, part number 1.) The existing `file_done.emit(str(out_path))` already carries each finished mp4 path — the upload step keys off that + the sidecar.

## Phase 3 — Thumbnail generator (Pillow)

**New `src/noveltrans/youtube/thumbnail.py`:**
- `render_thumbnail(base_image, out_path, *, title, part_number, tagline, font_file, width=1280, height=720) -> Path`.
- Steps: `_cover_fit` the base image to 1280×720 (reuse the pattern from `player_skin._cover_fit`); optional subtle dark scrim/gradient for text legibility; top-left multi-line title with word-wrap (`_wrap_text` measuring with `ImageFont.getbbox`); bottom-center `PHẦN {N}` large + tagline line beneath.
- `_glow_text(draw_layer, xy, text, font, ...)` helper: render text to a transparent layer, composite a blurred tinted copy (outer glow) via `ImageFilter.GaussianBlur`, then a dark stroke (`stroke_width`/`stroke_fill` in `ImageDraw.text`) + light fill — approximating the example's glow.
- Load fonts via `font_dir_context()`; default to `PlayfairDisplay-Regular.ttf`, honoring the 023 font selection (map the selected `VIDEO_FONTS` key → its `file`).
- Pure enough to unit-test: assert it writes a 1280×720 file ≤ the YouTube 2 MB thumbnail limit and that `_wrap_text`/timestamp helpers behave. Save as JPEG (quality tuned) if PNG would exceed 2 MB.

**GUI:** add a thumbnail base-image picker (new `AppConfig.youtube_thumbnail_image`), defaulting to the existing `video_image_path`. A "tagline" text field feeds the bottom subtitle line (the "…" line); default from `translated_description` first sentence or blank.

## Phase 4 — YouTube OAuth + upload client (I/O)

**New `src/noveltrans/youtube/oauth.py`:**
- `class YouTubeAuth` — loads/saves the token JSON; `ensure_access_token()` refreshes via the token endpoint (`requests.post`) using the stored refresh token; `run_interactive_flow(client_id, client_secret)` spins up a localhost `http.server` on an ephemeral port, opens the consent URL (`webbrowser.open`), captures the `code` on the loopback redirect, exchanges it for tokens, persists them. Raises a typed `YouTubeError` (new, in `errors.py`) on failure.

**New `src/noveltrans/youtube/upload.py`:**
- `upload_video(auth, *, video_path, title, description, privacy, category_id="22", playlist_id=None, thumbnail_path=None, cancelled=None, progress=None) -> str` (returns the YouTube videoId).
- **Resumable upload** over `requests`: initiate a resumable session (`POST .../videos?uploadType=resumable&part=snippet,status`), then `PUT` the file in chunks, honoring `cancelled()` between chunks and reporting `progress(bytes_done, total)`. On 5xx/`308`-incomplete, retry with exponential backoff (respect `Retry-After`); resume from the last-confirmed byte using the `Range` response header.
- After success: `thumbnails.set` (multipart PUT of the thumbnail, ~50 units) and, if `playlist_id`, `playlistItems.insert`.
- All user-facing error strings are Vietnamese, consistent with `TtsError`.

**New `src/noveltrans/errors.py` addition:** `class YouTubeError(Exception)` (mirroring `TtsError`).

## Phase 5 — Upload worker + idempotency + GUI wiring

**New `YouTubeUploadWorker(QThread)` in `src/noveltrans/gui/workers.py`** (structural clone of `VideoWorker`):
- Inputs: project path, list of produced mp4 paths (or it re-discovers `video_dir/*.mp4`), privacy, playlist, thumbnail base image, tagline, font key.
- For each mp4 in order: read its `<mp4>.json` manifest; build title/description (Phase 1) from `project.meta` + manifest + `project.counts()["total"]`; render the thumbnail (Phase 2/3) to a temp PNG; call `upload_video`; on success record `{filename: videoId}` in an **upload manifest** `video_dir/.youtube-uploads.json`.
- **Idempotency/resumability:** before uploading, skip any mp4 already present in `.youtube-uploads.json` (don't re-spend ~1600 units). Signals: `progress(done, total, name)`, `bytes_progress(...)`, `file_done(path, video_id)`, `finished_ok(count)`, `failed(msg)` — matching existing patterns; register with `track_worker(...)` for keep-awake.
- Cancel support (`self._cancelled`, checked between parts and between upload chunks), same as `VideoWorker`.

**Modify `src/noveltrans/gui/tab_audio.py`:**
- In `_build_video_box`: add privacy `QComboBox` (Công khai/Không công khai/Riêng tư), a "Tải lên YouTube sau khi tạo" `QCheckBox`, an optional playlist field, a thumbnail base-image picker + tagline field, and a standalone **"Tải lên YouTube"** button (uploads existing rendered parts).
- In `_start_video` / `_on_video_finished`: when the auto-upload checkbox is set, chain into `YouTubeUploadWorker` after `finished_ok`. Standalone button starts the worker directly against `video_dir`.
- Add first-run auth handling: if no valid token, prompt the user (Vietnamese `QMessageBox`) to complete Google consent; if `youtube_client_id`/`secret` are unset, direct them to Settings with setup instructions. Disable the upload button with a tooltip when client credentials are missing.
- Reuse the existing worker lifecycle helpers (`_reset_video_ui`, `shutdown`, `has_running_workers`) — extend them to also cover `_youtube_worker`.

**Modify `src/noveltrans/config.py`:** add properties `youtube_client_id`, `youtube_client_secret`, `youtube_privacy` (default `"unlisted"`), `youtube_playlist_id`, `youtube_thumbnail_image`, `youtube_autoupload_enabled` (bool), following the existing one-property-per-setting pattern. Add corresponding fields to the Settings dialog.

## Phase 6 — Packaging / deps

- **No new runtime dependency** if the hand-rolled loopback OAuth + `requests` upload path is taken (`requests`, `pillow` already present). This is the recommended path and keeps `NovelTrans.spec` untouched.
- If instead the Google client libraries are adopted: add `google-api-python-client` + `google-auth-oauthlib` to `pyproject.toml` and extend `NovelTrans.spec` — add a `collect_all("googleapiclient")` (it ships a `discovery` cache and data files PyInstaller misses) and hidden imports for `google.auth`, `google_auth_oauthlib`. Prefer avoiding this.
- No new package-data needed unless a decorative display font is bundled (then add its `*.ttf` — `package-data` already globs `tts/assets/*.ttf`; if the thumbnail font lives under `youtube/assets`, add that glob).
- Verify `webbrowser.open` works from the frozen `.app` (it does; it shells out to the OS default browser).

Add `tests/test_youtube_metadata.py` and `tests/test_youtube_thumbnail.py` (pure builders only, no network/ffmpeg — matching the `test_video.py`/`test_merge.py` convention):

## Quota / safety notes

- A `videos.insert` costs **~1600 units**; default quota is **10,000/day ≈ 6 uploads/day**. `thumbnails.set` ~50, `playlistItems.insert` ~50. The UI should warn before a batch and show remaining-capacity guidance ("YouTube giới hạn ~6 video/ngày"). Consider a per-run cap and a clear message when the API returns `quotaExceeded`.
- **Idempotency** via `.youtube-uploads.json`: never re-upload a part that already has a recorded videoId; a failed/cancelled batch resumes at the first not-yet-uploaded part.
- **Resumable upload** with chunked PUT + exponential backoff on 5xx/`308`, honoring `Retry-After`, so a large multi-GB part survives a transient network drop without restarting.
- **Cancellation** checked between parts and between upload chunks, mirroring the ffmpeg poll/cancel pattern.
- Credentials are stored plaintext (QSettings / token JSON) — same posture as the existing Claude key/cookies; document this in `config.py`'s module docstring note.

## Risks & follow-ups

- **User must create a Google Cloud project + OAuth Desktop client and (for public uploads) may need to pass Google's API verification / add themselves as a test user** while the OAuth consent screen is unverified (unverified apps are limited to test users and tokens expire in 7 days). This is the single biggest user-facing setup burden — document it thoroughly in Vietnamese. Requires user input: `client_id`, `client_secret`, and consenting in-browser.
- **Vietnamese author** may be missing for existing projects (new field) — fallback handled, but back-fill is manual.
- **Decorative font fidelity:** the approximated glow won't perfectly match the example's script face; bundling a Vietnamese-covering display font is the follow-up if fidelity matters.
- **7-day refresh-token expiry** for unverified OAuth apps means periodic re-consent; note in UI.
- **Scheduled publishing / `publishAt`** and richer playlist management are deferred to a follow-up.
