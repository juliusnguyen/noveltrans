# Feature 020 — VIDEO-SOUNDBAR: waveform visualizer + faded chapter titles

## Summary

Feature 019 (`src/noveltrans/tts/video.py`) renders a still background image + concatenated chapter audio, with the novel title (top) and chapter title (bottom) burned in via ASS, encoded with `-tune stillimage`. Feature 020 makes the video "nicer" by:

1. Overlaying an **audio-driven waveform** (`showwaves`, `mode=cline`, cyan) in the middle of the frame, on top of 019's blurred-cover + sharp-fit background.
2. Moving the **chapter title to just below the waveform** (was bottom-anchored).
3. **Fading the chapter title** in/out on each chapter change via ASS `\fad`.
4. Because the frame now animates every frame, **dropping `-tune stillimage`**, using a normal x264 preset, and **bumping fps 12→25** for a smooth wave.

The novel title (persistent, top), the concatenated AAC audio, `-shortest`, `_with_real_durations`, the YouTube description `.txt`, chapter timing, and the blurred-fill background are all **unchanged**. The change is confined to three functions in `video.py` (`_filtergraph`, `render_video`, `build_ass_subtitles`), two default-fps bumps, and two GUI cosmetic strings. Always-on (no new GUI control, no config).

---

## The audio-both-filtered-and-copied question (the crux) — RESOLVED

**It works with `-c:a copy` and no `asplit`.** ffmpeg permits a single input stream (`1:a`) to be consumed by `filter_complex` (`[1:a]showwaves…`) **and** simultaneously mapped to an output with `-map 1:a -c:a copy`. The demuxed packets feed the copy path; a separate decode feeds `showwaves`. These are independent consumers of the same input stream (the "stream already used" error only fires when the *same stream is mapped to output twice*, which we do not do). `asplit` is irrelevant — it splits an already-*filtered* audio stream into multiple *filter* branches.

So: keep `audio.m4a` as input 1, keep `-map 1:a -c:a copy`, and additionally tap `[1:a]` in the filtergraph. No re-encode in the render pass. Fallback if any ffmpeg build ever refuses: `-c:a aac -b:a 96k` in this pass (cheap; audio is already AAC). **This was verified with a real render** (see history de-risk: pre-encoded `audio.m4a` fed to showwaves + `-c:a copy` → MP4 with working video+audio streams).

## New ffmpeg render command (step 2 of `render_video`)

Input order stays: image first (`-loop 1 -i image`) → `[0:v]`; audio second (`-i audio.m4a`) → `[1:a]` / `1:a`.

```
ffmpeg -y \
  -loop 1 -framerate <fps> -i <image> \
  -i <audio.m4a> \
  -filter_complex "<NEW GRAPH>" \
  -map "[v]" -map 1:a \
  -c:v libx264 -preset veryfast -pix_fmt yuv420p -r <fps> \
  -c:a copy -shortest <out.mp4>
```

Changes vs 019: `[1:a]showwaves…` added to the graph and overlaid; `-tune stillimage` **removed**; `-preset veryfast` added. Everything else (`-map [v]`, `-map 1:a`, `-c:a copy`, `-shortest`, `-pix_fmt yuv420p`) identical.

### New `_filtergraph` — extend to take audio + build the waveform

`_filtergraph` currently takes `(width, height, subs_path, font_dir)`. Give it `fps` and produce:

```
[0:v]scale=W:H:force_original_aspect_ratio=increase,crop=W:H,boxblur=20:2,eq=brightness=-0.25[bg];
[0:v]scale=W:H:force_original_aspect_ratio=decrease[fg];
[bg][fg]overlay=(W-w)/2:(H-h)/2[base];
[1:a]showwaves=s=<WW>x<WH>:mode=cline:colors=aqua:rate=<fps>:draw=full[viz];
[base][viz]overlay=(W-w)/2:<VY>,subtitles='<subs>':fontsdir='<fonts>'[v]
```

Where (scaled to the frame so it also renders for the 640x360 test):
- `WW = int(width * 0.9)` (1728 at 1920), `WH = int(height * 0.18)` (194 at 1080)
- `VY = int(height * 0.30)` (324 at 1080) — wave band spans ~y=324..518, centered X.
- `rate = fps` so the wave advances one column-group per output frame (matches `-r fps`).

`_filtergraph` now needs `fps`: `_filtergraph(width, height, fps, subs_path, font_dir)`. The `[1:a]` label is a fixed input reference; the function stays pure/string-only (unit-testable by asserting substrings).

### Waveform styling (fixed tasteful default, no user control)
- `mode=cline` — filled center line (the decided look), a clean band.
- `colors=aqua` (cyan) — matches the verified prototype; single fixed color.
- `draw=full` — smoother fill (safe default).
- Transparent background by default → composites cleanly over `[base]` via `overlay` (verified). Keep `-pix_fmt yuv420p` on the final encode for player compatibility.

## ASS builder changes (`build_ass_subtitles` + `_ASS_STYLES`)

Two changes, both in the chapter path; **Novel event unchanged** (persistent, un-faded, top, Alignment 8).

**1. Chapter placement below the wave.** In `_ASS_STYLES`, the `Chapter` style is `Alignment 2` (bottom-center), `MarginV 120`. Change to **`Alignment 8`** (top-center anchored) so `MarginV` measures from the top, and set `MarginV` to clear the wave band. Make the chapter `MarginV` a computed value passed at format time (like `nsize`/`csize`): `chapter_margin_v = int(height * 0.52)` (≈562 at 1080) so the chapter text sits just under the wave bottom (~518). Add a `chapter_margin_v: int | None = None` parameter to `build_ass_subtitles` (default derived from height) and thread it into `_ASS_STYLES.format(...)`. Novel style keeps Alignment 8 / MarginV 60.

- Tradeoff to note: exact `MarginV` and the `novel_font_px`/`chapter_font_px` are eyeball values — a real 1920x1080 render is needed to confirm the wave, title, and chapter don't collide. The fractions above are the starting point.

**2. `\fad` on each chapter event.** At the per-segment `Dialogue` line (currently `…,,{_escape_ass(seg.title)}`), prepend a fade override **outside** the escaped title:

```python
text = "{\\fad(400,400)}" + _escape_ass(seg.title)
```

`_escape_ass` only touches the *title* (it turns `{`/`}` into `(`/`)`), and the `\fad` block is concatenated after escaping, so the override is never corrupted and a malicious title can't break out of it. 400 ms in / 400 ms out is a good default for chapters that are minutes long.

Keep the Novel `Dialogue` line exactly as-is (no `\fad` → stays solid the whole video).

## Encoder change + cost writeup

| 019 (static) | 020 (motion) |
|---|---|
| `-tune stillimage` | **removed** |
| (no preset → x264 default `medium`) | **`-preset veryfast`** |
| `fps=12` | **`fps=25`** |
| video stream ~free (one repeated frame) | full motion-encoded stream |

- **Remove `-tune stillimage`**: optimizes for a non-changing frame; wrong with a wave animating every frame.
- **fps 12→25**: change the default in `render_video` (line ~276) and `VideoWorker.__init__` (workers.py line ~753). The wave looks noticeably smoother at 25; `showwaves rate` follows `fps`. Tradeoff: ~2x frames vs 019.
- **`-preset veryfast`**: keeps encode time bounded on long novels; note `medium` as a quality/size alternative.
- **Cost:** 019's video stream was near-zero bytes (static). 020 is genuine motion video, so **files are meaningfully larger and encoding is much slower** — roughly proportional to duration and frame count. The main regression to surface.
- **Deadline:** `render_video` sets `deadline = max(3600, int(total)*3)`. Motion encode at veryfast is faster than realtime but the `*3` headroom is thinner; bump to `max(3600, int(total)*6)` to avoid spurious timeouts on long batches.

## Risks / uncertainties

- **Audio filtered + copied (main risk):** resolved as supported above and verified with a real render; the real-render test is the guardrail — fallback is `-c:a aac -b:a 96k` in the render pass.
- **Font / position tuning:** the wave `s=`, overlay `VY`, chapter `MarginV`, and font sizes are eyeballed fractions; only a real 1920x1080 render confirms the novel title, wave band, and chapter title don't overlap. Needs a visual pass.
- **Encode cost/size:** genuine regression vs 019's near-free static stream — surfaced in the confirm dialog; `veryfast` chosen to bound time.
- **Fade duration:** 400/400 ms is a judgment call; fine for minute-scale chapters, trivially tunable.
- **CJK caveat unchanged:** Vietnamese chapter titles render; original-language (CJK) still tofu — same as 019.
