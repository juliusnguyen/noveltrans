## [DE-RISK] The main technical risk resolved before planning — 2026-07-18
The crux: can the audio stream feed the `showwaves` filter AND be mapped/copied to the
output at the same time? **Yes** — proven with real ffmpeg:
- A pre-encoded `audio.m4a` fed to `[1:a]showwaves…[viz]` while `-map 1:a -c:a copy`
  produced an MP4 with **both** a working h264 video and aac audio stream. `-c:a copy`
  works even though the same stream is filtered — ffmpeg splits the input internally.
So the 019 audio pipeline (concat → `audio.m4a` → `-c:a copy`) needs **no change**; the
render step just adds the waveform overlay and drops `-tune stillimage`.

Prototype filtergraph that rendered cleanly (to build from):
`[1:a]showwaves=s=1000x200:mode=cline:colors=aqua:rate=25[viz];`
`[0:v]<019 blurred-fill>[bg];[bg][viz]overlay=(W-w)/2:H*0.32,subtitles=…[v]`
→ `-map [v] -map 1:a -c:a copy -shortest`. (Drop `-tune stillimage`; the wave animates.)

Cost note surfaced: the video is no longer static, so encode is slower / file bigger than
019's near-free still stream — the plan must account for this (fps ~25, a normal preset,
a bumped deadline, and the confirm-dialog warning).
