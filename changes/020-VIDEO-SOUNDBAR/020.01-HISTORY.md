## [DE-RISK] The main technical risk resolved before planning — 2026-07-18
The crux: can the audio stream feed the `showwaves` filter AND be mapped/copied to the
output at the same time? **Yes** — proven with real ffmpeg:
- A pre-encoded `audio.m4a` fed to `[1:a]showwaves…[viz]` while `-map 1:a -c:a copy`
  produced an MP4 with **both** a working h264 video and aac audio stream. `-c:a copy`
  works even though the same stream is filtered — ffmpeg splits the input internally.
So the 019 audio pipeline (concat → `audio.m4a` → `-c:a copy`) needs **no change**; the
render step just adds the waveform overlay and drops `-tune stillimage`.

Prototype filtergraph that rendered cleanly (to build from):
`[1:a]showwaves=s=1000x200:mode=cline:colors=aqua:rate=25[viz];`
`[0:v]<019 blurred-fill>[bg];[bg][viz]overlay=(W-w)/2:H*0.32,subtitles=…[v]`
→ `-map [v] -map 1:a -c:a copy -shortest`. (Drop `-tune stillimage`; the wave animates.)

Cost note surfaced: the video is no longer static, so encode is slower / file bigger than
019's near-free still stream — the plan must account for this (fps ~25, a normal preset,
a bumped deadline, and the confirm-dialog warning).

## [REVISION] "Pro music player" skin + bigger full-width bars — 2026-07-18
User's first bars/split render read as flat ("not good enough") and sent image #3 — a
polished music-player template: pastel gradient, the photo framed with rounded corners +
a soft shadow on the LEFT, a record-player widget (vinyl + rainbow + tonearm, seek bar,
play/prev/next buttons) on the RIGHT, and a wide waveform along the bottom. Asked to "make
it like a pro player tool" and "make the sound bar bigger".

**Key realisation:** ffmpeg can't draw gradients, rounded corners, drop shadows, or a
vinyl record cleanly — but **Pillow can**. So the static chrome is now baked once per
render as a PNG and ffmpeg only animates the parts that move (bars + titles) on top.

- **New `tts/player_skin.py`** — `build_player_skin(image, out, width, height)` draws the
  full-frame skin with Pillow: vertical pastel gradient (lavender→pink→blue) + two soft
  colour glows; the photo cover-fit into a rounded box on the left with a blurred drop
  shadow + white frame; the vinyl widget (black disc, rainbow grooves on the right arc,
  rainbow centre label, spindle, blue tonearm); a seek bar with a knob; play/prev/next
  buttons. Unreadable image → placeholder card (never crashes the render).
- **`PlayerLayout.of(w,h)`** — one source of truth for the coords that must agree across
  modules (bars rect → `_filtergraph`; title margins/fonts → `build_ass_subtitles`). Pure
  arithmetic, unit-tested; everything scales with resolution.
- **`_filtergraph`** simplified — input 0 is now the baked skin, so it only overlays the
  **bigger** bars: `showfreqs` **full-width** (~95%) and tall (~19% H) along the **bottom**,
  coloured **purple** (`0x8a52c8`) so they read on the light skin (white would vanish),
  then burns the titles. No more in-ffmpeg photo/blur/split.
- **`build_ass_subtitles`** — titles are now the player's "now playing" block in the right
  column: Novel = muted grey-purple album line, Chapter = bold dark-purple track line just
  below it (still `\fad`s on change). Light-theme styles (no dark scrim; soft shadow).
- **`render_video`** bakes the skin to a temp PNG and feeds it as input 0.
- **pyproject** — added `pillow>=10.0` (was only transitive). **tab_audio** box title →
  "trình phát nhạc: ảnh + cột sóng + tên chương".
- Tests: `PlayerLayout` scales + places photo-left/bars-bottom; `build_player_skin` emits
  a right-sized PNG and survives an unreadable image; `_filtergraph` asserts full-width
  purple bars anchored bottom-left over `[0:v]`; ASS margins read from `PlayerLayout`.
