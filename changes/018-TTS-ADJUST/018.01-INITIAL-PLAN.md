# Implementation Plan: Adjustable TTS Settings (4 knobs)

## Summary

Add four Settings-dialog controls that shape TTS output: **paragraph gap**, **speaking speed**, **volume/gain**, and **expressiveness (temperature)**. Three of them (gap, volume, temperature) apply at or around the single chokepoint `TtsEngine.synthesize_chapter`; the fourth (speed) is an ffmpeg `atempo` post-process in the worker, sitting right beside the existing mp3 conversion. Values live in `AppConfig` (QSettings) with defaults that reproduce today's output byte-for-byte, are edited in `SettingsDialog`, read in `AudioTab`, and threaded into `AudioWorker` exactly the way `clean_text` was in feature 017. This is a superset of the clean-text feature; the only genuinely non-trivial part is speed (atempo chaining, duration recompute, ffmpeg gating).

Verified anchors: `paragraph_gap_seconds` is `base.py:51`; `synthesize_chapter` def at `base.py:69` (gap built at `:96`, `audio = np.concatenate` at `:105`, `save_wav` `:107`, `return len(audio)/self.sample_rate` `:108`). `VieneuEngine.__init__` at `vieneu.py:49`, `synthesize` at `vieneu.py:109` (`infer(text, voice=...)` `:113`, `infer(text)` `:114`). `get_tts_engine(name, *, voice="")` at `__init__.py:11`. `convert_to_mp3` at `convert.py:32` (ffmpeg argv `:37`). `AudioWorker.__init__` at `workers.py:333` (fields `:346-354`); `synthesize_chapter` called at `:429` (sequential) and `:493` (parallel `_synth_one`); mp3 convert at `:437-440` and `:501-504`; probe engine built at `:367`, per-thread engine at `:475`. ## The four knobs

### 1. Paragraph gap (silence between chunks)
- **Where it applies:** inside `synthesize_chapter`, `base.py:96` — the `gap = np.zeros(int(self.sample_rate * self.paragraph_gap_seconds), ...)` line.
- **Change:** add a parameter to the signature rather than mutating the class attribute (cleaner under parallel workers, and keeps the class default). New signature:
  `synthesize_chapter(self, title, body, out_path, cancelled=None, clean=True, clean_extra_remove="", gap_seconds: float | None = None, volume: float = 1.0)`.
  Inside: `gap_len = self.paragraph_gap_seconds if gap_seconds is None else gap_seconds` and build `gap` from `gap_len`.
- **Default:** `None` → falls back to `self.paragraph_gap_seconds` (0.4). Config default 0.4.
- **Range:** 0.0–2.0 s.

### 2. Volume / gain
- **Where it applies:** same `synthesize_chapter`, on the final `audio` right after `np.concatenate` (`base.py:105`), before `save_wav`. Final-buffer multiply (not per-chunk) is cheaper and gaps are zeros so unaffected.
- **Change:** `if volume != 1.0: audio = np.clip(audio * volume, -1.0, 1.0)`. **Recommend linear multiplier, not dB** (simpler, matches the "×" mental model), and **hard-clip to [-1, 1]** to prevent wraparound distortion on gains > 1.0. Note in a tooltip that high gain can clip.
- **Default:** 1.0. **Range:** 0.1–3.0×. numpy only — **no ffmpeg**, so it works for WAV too.

### 3. Speed (the risky 20%) — ffmpeg `atempo` post-process
- **Where it applies:** in `AudioWorker`, as a post-process on the rendered WAV, **after** `synthesize_chapter` returns and **before** `convert_to_mp3`. It runs on the whole file so the paragraph gaps get sped up too (confirmed: post-process the whole rendered file, not per-chunk).
- **New function in `convert.py`:**
  - `_atempo_filters(tempo: float) -> list[str]` — decompose any positive tempo into factors each within `[0.5, 2.0]`: repeatedly divide by 2.0 while > 2.0, multiply by 2.0 (i.e. emit 0.5) while < 0.5, then append the remainder. e.g. `2.5 → [2.0, 1.25]`, `0.25 → [0.5, 0.5]`, `1.5 → [1.5]`. Within the chosen 0.5–2.0 slider range this is always a single filter, but implement/test it generally so widening the range later is safe.
  - `apply_tempo(wav_path: Path, tempo: float) -> Path` — if `tempo == 1.0`, return `wav_path` untouched (no ffmpeg). Otherwise run `ffmpeg -y -i wav -filter:a "atempo=...,atempo=..." <tmp.wav>`, then `os.replace` the temp over the original; return `wav_path`. Mirror `convert_to_mp3`'s error handling exactly (FileNotFoundError → the "Không tìm thấy ffmpeg…" TtsError; TimeoutExpired; nonzero returncode → stderr tail).
- **Duration recompute:** `synthesize_chapter` returns pre-speed duration. `atempo` is an exact time-scale, so after `apply_tempo` set `seconds = seconds / tempo` in the worker (no ffprobe needed). This matters — `audio_seconds` feeds m4b chapter markers in `MergeWorker`.
- **WAV-needs-ffmpeg gating:** WAV normally needs no ffmpeg, but `atempo` does. **Gate the speed control in `SettingsDialog` on `ffmpeg_available()`** (disable + tooltip, mirroring the tab's mp3 gating at `tab_audio.py:62-66`). Defensively, the worker should only call `apply_tempo` when `speed != 1.0 and ffmpeg_available()`; if speed is set but ffmpeg is missing (e.g. removed after the setting was saved), skip the tempo pass, leave `seconds` unscaled, and emit a one-time progress notice ("Bỏ qua tốc độ: cần ffmpeg") rather than failing every chapter.
- **One-pass vs two-pass:** **Recommend two passes** (`apply_tempo` writes WAV, then `convert_to_mp3` reads it) — it reuses `convert_to_mp3` untouched, keeps the worker branching trivial, and the extra decode/encode is negligible against ~seconds of neural synthesis; atempo→WAV is lossless so no quality loss. A combined single ffmpeg call (atempo filter + mp3 encode in one invocation) is possible and saves one pass, but duplicates `convert_to_mp3`'s logic and complicates the mp3/wav branch — note it as a future optimization, not now.
- **Default:** 1.0. **Range:** 0.5–2.0×.

### 4. Expressiveness (temperature)
- **Where it applies:** threaded into `VieneuEngine.synthesize` → `self._tts.infer(...)` at `vieneu.py:113-114`.
- **Change:**
  - `get_tts_engine(name, *, voice="", temperature: float | None = None)` (`__init__.py:11`), passing `VieneuEngine(voice=voice, temperature=temperature)`.
  - `VieneuEngine.__init__(self, voice="", temperature: float | None = None)` stores `self.temperature`.
  - `synthesize`: build kwargs and only include temperature when set:
    ```
    kwargs = {}
    if self.voice: kwargs["voice"] = self.voice
    if self.temperature is not None: kwargs["temperature"] = self.temperature
    return tts.infer(text, **kwargs)
    ```
    This keeps the two existing assertions green (`infer("xin chào", voice="Xuân Vĩnh")` and `infer("xin chào")`) because default `None` adds no kwarg.
- **Default: pass nothing (recommended).** Use a sentinel `0.0` in config meaning "unset"; the worker converts `temperature if temperature > 0 else None`. Passing nothing is the only way to guarantee byte-for-byte parity with today (the current code passes no temperature, so `infer` uses whatever the installed build's default is — 0.4 for v3turbo — which we should not hard-code, in case the wrapper's default differs). **Range:** 0.0–1.5 (0.0 shown as "Mặc định").
- **Both engine-construction sites must pass temperature:** the probe at `workers.py:367` **and** the per-thread engine at `workers.py:475` (the probe is reused as the first pool thread's engine, so missing either leaves some threads at the wrong temperature). `TtsVoicesWorker` (`workers.py:691`) lists voices only — leave it at the default.

- `DEFAULT_TTS_SPEED = 1.0`

Use `type=float` on `self._s.value(...)` and `max(lo, min(hi, ...))`.

## Testing strategy (model-free + ffmpeg-free)

Extend `FakeTtsEngine` to capture the saved samples (store the array in `save_wav`, not just the path) so gain is assertable.

- **Gap:** `synthesize_chapter(..., gap_seconds=0.3)` on `FakeTtsEngine` (1000 Hz) → gaps of 300 samples each; assert returned duration matches `(chars + gaps)/1000`. Contrast with default to prove the knob moves duration.
- **Volume:** engine returns ones; `volume=2.0` → saved samples clipped to 1.0 (tests the clamp); `volume=0.5` → 0.5. Assert on the captured array.
- **Temperature threading:** in `TestVieneuEngine`, build via `get_tts_engine("vieneu", voice=..., temperature=0.7)`, assert `infer` called with `temperature=0.7`; and default construction → `infer` called with **no** temperature kwarg (existing `test_synthesize_passes_voice`/`_default_voice` already cover parity).
- **`_atempo_filters`:** `2.5 → [2.0, 1.25]`, `0.25 → [0.5, 0.5]`, `1.5 → [1.5]`, `1.0 → [1.0]`.
- **`apply_tempo` argv:** patch `convert.subprocess.run`, assert the `-filter:a` arg contains the chained `atempo=...` string for a >2.0 tempo; assert `tempo == 1.0` makes **no** subprocess call and returns the path unchanged. (Model of the existing `test_convert_replaces_wav_with_mp3`.)
- **Speed duration rescale:** worker test with `speed=2.0`, patch `get_tts_engine → FakeTtsEngine` and patch `apply_tempo` to a no-op; assert stored `audio_seconds` is exactly half the base duration.
- **ffmpeg gating:** worker with `speed=1.5` and `convert.ffmpeg_available` patched `False`; assert `apply_tempo` not called and `seconds` unscaled.
- **Config:** `TestConfigTts` mirroring `TestConfigWorkers` — defaults (0.4 / 1.0 / 1.0 / 0.0), clamps at each bound, round-trip.
- **Defaults reproduce current output (explicit):** run `AudioWorker` with all-default new params; assert `apply_tempo`/`subprocess.run` never called, `synthesize_chapter` receives `gap_seconds=None`/`volume=1.0`, and `get_tts_engine` receives `temperature=None` — i.e. identical behavior to pre-feature.

Model tests stay `@pytest.mark.live`; nothing above needs the 334 MB model or a real ffmpeg.

## Risks / things to flag

- **temperature default is a genuine unknown:** we deliberately pass nothing when unset, because the installed build's default (0.4 for v3turbo) is not something we should hard-code — the "pass nothing" path is the only guaranteed parity. Flagged.
- **Volume hard-clip** prevents wraparound but can audibly distort at high gain; acceptable for a user-driven "×" control, noted in the tooltip. A true normalize would be more work for little gain.
- **Duration scaling** assumes `atempo` is a pure time-scale (it is); gaps are inside the file so they scale correctly.
- **Two engine-construction sites** — easy to update the probe (`:367`) and miss the per-thread build (`:475`); both must carry temperature.
- **Two-pass ffmpeg** (tempo then mp3) is intentional for simplicity; combined single-pass is a possible later optimization.
- **`setSpecialValueText`** must pair with `minimum == 0.0` so the "Mặc định" sentinel maps cleanly to "unset".

## Note for the user: there is no "steps / accuracy" knob

This VieNeu build is **v3turbo**, an autoregressive LM-based TTS (LMDeploy backbone, `temperature`/`top_k` sampling) — **not** a diffusion model. There is no NFE / steps / `num_inference_steps` parameter to expose; that concept belongs to diffusion-TTS, which this isn't. The nearest lever is `top_k` (sampling breadth), but it's a subtle quality/variety dial, not an "accuracy" control, and would confuse more than it helps — **recommend leaving it out.** The four knobs above (gap, speed, volume, temperature) are the real, available controls.
