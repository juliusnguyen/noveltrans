## [DECISIONS] Scope settled before planning — 2026-07-18
User chose all four adjustables, plus asked for a "steps/accuracy" control.

### Four confirmed knobs (verified to exist)
1. **Pause length** — `paragraph_gap_seconds` (0.4, hardcoded in `tts/base.py`).
2. **Speaking speed** — NOT native to VieNeu; needs ffmpeg `atempo` post-processing.
3. **Volume/gain** — numpy scalar multiply on the samples.
4. **Expressiveness (temperature)** — VieNeu `infer`'s `temperature` param.

### The "steps" knob does NOT exist — corrected with the user
The user remembered a "more steps = more accurate" control. Checked the installed
VieNeu source: this is the **`v3turbo`** build — an **autoregressive LM TTS** (LMDeploy
backbone, `temperature`/`top_k` sampling), **not diffusion**. There is no
NFE/steps/num_inference_steps parameter anywhere in `infer`. That control belongs to
diffusion-TTS (e.g. F5-TTS), which this isn't. Told the user plainly rather than build a
slider that does nothing. `top_k` is the nearest lever but it's not an accuracy dial —
recommended leaving it out.

### Verified pipeline facts
- `synthesize_chapter` (`tts/base.py`) is the single chokepoint (both AudioWorker paths
  + CLI). Pause + volume apply here; temperature threads into `vieneu.py::synthesize`'s
  `infer(...)`; speed is a post-process (ffmpeg atempo) like the mp3 conversion.
- `convert.py` already runs ffmpeg for mp3 — atempo belongs there. Constraints: atempo
  range 0.5–2.0 (chain beyond), speed changes duration (recompute), and WAV output would
  now need ffmpeg for speed (gate the control on `ffmpeg_available()`, as the mp3 option
  already is).
- Defaults must reproduce today's output (gap 0.4, speed 1.0, volume 1.0, temperature =
  pass-nothing for exact parity).

## [STEP 4] Speed via ffmpeg atempo — 2026-07-18
Plan 018.01 Step 4. `tts/convert.py`; tests in `tests/test_tts.py`.

- **`_atempo_filters(tempo)`** — decomposes any positive tempo into factors each within
  ffmpeg's [0.5, 2.0] atempo range, product = tempo (2.5→[2.0,1.25], 0.25→[0.5,0.5]).
  Within the 0.5–2.0 slider it's always one factor, but the general form keeps a wider
  range safe to add later.
- **`apply_tempo(wav_path, tempo)`** — `tempo==1.0` is a no-op (no ffmpeg call); else runs
  `ffmpeg -filter:a atempo=…` into a temp file that replaces the original in place. Mirrors
  `convert_to_mp3`'s error handling; cleans up the temp file on failure so a botched run
  leaves the original WAV untouched.

### Tests: 9 new (427 total)
`_atempo_filters` parametrized (6 cases, all factors in range); `apply_tempo` 1.0 no-op
(asserts subprocess never called); the chained-filter argv for 2.5× (`atempo=2,atempo=1.25`
as one `-filter:a` arg); and the error path (temp cleaned up, original intact).

### Verified against REAL ffmpeg (not just mocked argv)
ffmpeg is present locally, so drove `apply_tempo` on a generated 2s tone:
- 2.0× → 1.002s (scales by exactly 1/tempo — so the worker can do `seconds /= tempo`
  without an ffprobe pass, which feeds the m4b chapter markers).
- 2.5× chained → 0.805s (≈2/2.5). The filter chaining works end to end.

Pitch preservation is atempo's defining property (not separately asserted here — it's the
filter's contract; the human listen-through before ship is the real check).

Verified: `427 passed, 5 deselected` (was 418); `ruff check` clean.

## [STEP 8] Model precision (int8/fp32) — a fifth knob — 2026-07-18
Added after the user asked which model VieNeu uses and how to raise quality.

### Investigation
Traced the model: our `Vieneu()` call defaults to the **v3turbo** build →
`pnnbao-ump/VieNeu-TTS-v3-Turbo` (48 kHz, ~0.3B autoregressive, ONNX/CPU int8 by
default). The one real quality lever the wrapper exposes is `precision`. Confirmed
against the **actual HF repo file tree** (`list_repo_files`): the ONNX graphs are
`onnx`, `onnx_int8`, `onnx_update` — so `precision` maps int8→onnx_int8, fp32→onnx_update,
and there is **no fp16/int4** (unknown values fall back to int8). Two meaningful tiers,
not more; told the user plainly.

### Wiring (mirrors temperature, but at model construction)
- `config.py`: `DEFAULT_TTS_PRECISION="int8"`, `TTS_PRECISIONS=("int8","fp32")`, and a
  `tts_precision` str property that **validates** — any non-member reads/writes back as
  int8 (so a hand-edited QSettings can't feed a bad subfolder to the model).
- `VieneuEngine.__init__` stores `precision`; `load()` calls `Vieneu(precision=self.precision)`.
- `get_tts_engine(..., precision="int8")` → engine.
- `AudioWorker` gains a `precision` field, passed to **both** engine builds.
- `tab_audio` passes `config.tts_precision`.
- `settings_dialog`: a "Chất lượng giọng" `QComboBox` (int8 fast / fp32 cao), saved in
  `accept()`, tooltip warning fp32 pulls a larger one-time download.

Default stays int8, so nothing changes unless chosen.

### Tests: 6 new (437 total)
Engine: `load()` passes `precision=fp32` to `Vieneu()`; default is int8. Config: default
int8, fp32 round-trips, `int4`→falls back to int8. Settings: dropdown loads/saves. Worker
defaults test now also asserts `precision="int8"`. Verified end to end: a real
`AudioWorker` with `precision="fp32"` reaches the actual `Vieneu(precision="fp32")` call.

Verified: `437 passed, 5 deselected` (was 433); `ruff check src tests` clean.

## [STEP 9] Split voice and style into two dropdowns — 2026-07-18
User asked to separate voice and style in the Audio tab.

### Investigation — style is a genuine, separate model axis
v3turbo's `infer` takes `style: str = "tu_nhien"` independently of `voice`. Extracted the
three real styles from the model's `voices_v3_turbo.json` presets: **`tu_nhien` (Tự
nhiên), `doc_truyen` (Kể chuyện), `tin_tuc` (Tin tức)**. The voice is the speaker
embedding; style is separate, so any voice × any style is valid.

**Key finding — the old labels lied.** The voice dropdown showed "… · Phong cách kể
chuyện", but the current code passes NO style to `infer`, so every voice was actually
read in `tu_nhien` (the infer default) regardless of its label. Confirmed the preset's
`style` field is only list metadata (v3turbo.py:116/206), never fed to inference (the
`style` param at :328 is). So this feature makes the style promise real for the first
time — and defaulting to `tu_nhien` reproduces today's output exactly.

**Interaction: fully independent** (user's choice) — two dropdowns, any combination,
style defaults to Tự nhiên.

### Implementation
- `config.py`: `DEFAULT_TTS_STYLE="tu_nhien"`, `TTS_STYLES` (ordered id/label), validated
  `tts_style` property (unknown → tu_nhien).
- `vieneu.py`: `VieneuEngine(style="")`; `synthesize` adds `style` to the infer kwargs
  only when set — so an empty style passes nothing (parity), a set one overrides.
- `get_tts_engine(..., style="")` → engine; `AudioWorker` `style` field → both engine builds.
- `tab_audio.py`: a "Phong cách" `QComboBox` next to voice; **voice labels stripped** of
  the "· Phong cách X" suffix (regex) since it's now its own control; style read/persisted
  on generate.

Consistent with the other 018 knobs, style applies to newly-rendered audio; "Tạo lại từ
đầu" re-renders existing chapters with the new style.

### Tests: 6 new (442 total)
Engine passes style when set / omits when empty (parity); config default+validation;
style combo lists the three and loads config; voice labels drop the suffix (id intact);
worker defaults test asserts `style=""`. Verified end to end: a real `AudioWorker` with
voice=Ngọc Linh + style=tin_tuc reaches `infer(voice="Ngọc Linh", style="tin_tuc")` — a
pairing the old single dropdown couldn't express.

Verified: `442 passed, 5 deselected` (was 437); `ruff check src tests` clean.
