# 031 — webtruyendich: crawl the full chapter list (lazy-loaded TOC)

## Bug

Scanning a webtruyendich novel misses most chapters — the app only picks up the
first ~135–870 instead of all ~1700+.

## Root cause (confirmed on the live TOC)

`/truyen/<slug>/danh-sach-chuong-day-du` **lazy-loads / infinite-scrolls** its
chapter list. A plain load renders only the first ~135 chapters and appends more
as the user scrolls to the bottom. The 029 scraper does `goto → read HTML` with
no scrolling, so `fetch_chapter_list` captures only whatever has loaded by that
instant.

Measured in a real browser:
- initial DOM: ~135 chapter links (max chapter 149)
- after scrolling to the bottom repeatedly: **1692 distinct chapters** (max
  1764), then stable — stops growing across 3 consecutive checks.

(The 029 test fixture was captured after a `sleep(2)` that happened to load a
fuller snapshot — 1722 — which hid the bug in tests.)

## Fix

Add a "scroll until the list stops growing" fetch to `WtdBrowserSession` and use
it for the TOC: scroll to the bottom, wait, and re-count the chapter links until
the count is stable for a few rounds (with a max-rounds cap), then read the HTML.
`parse_chapter_list` is unchanged.

## Scope

- `webtruyendich_browser.py` — `get_html` gains an optional lazy-scroll mode (a
  `scroll_item_selector`) that scrolls to the bottom until the selector count
  stabilises before returning the rendered HTML.
- `scrapers/webtruyendich.py` — `fetch_chapter_list` fetches the TOC with that
  scroll mode (item selector = the chapter-link selector).
- Tests: adapter wiring still passes the right URL and now the scroll selector;
  parser unchanged. Browser scrolling itself isn't unit-tested (same as the rest
  of the browser path).

## Out of scope

- Landing/metadata and chapter pages don't lazy-load; they keep the plain fetch.
