# 007 — Multiple Windows (Workspace Tabs) — Initial Implementation Plan

## Goal

Run 2+ independent crawl/translate/TTS flows in parallel inside one app window via an outer "workspace" tab bar. Each workspace is a full instance of the current four-tab stack with its own project and its own QThread workers. Single-flow behaviour is preserved as "one workspace on launch".

## Design in one sentence

Extract the inner `QTabWidget` + four tabs + all cross-tab wiring from `MainWindow` into a new `Workspace(QWidget)`; turn `MainWindow` into an outer `QTabWidget` host of `Workspace`s that keeps only app-global concerns (Settings, Library, geometry, dock badge, the single shared `AppState`, and the cross-workspace same-project guard).

---

## 1. Responsibility split: MainWindow vs Workspace

### Moves OUT of `MainWindow` into `Workspace` (new `src/noveltrans/gui/workspace.py`)

Everything currently in `MainWindow.__init__` lines 46–76 plus `_on_project_touched`:

- Construction of `self.tabs = QTabWidget()` (the inner four-tab stack) and the four tabs (`ScrapeTab`, `TranslateTab`, `ExportTab`, `AudioTab`), including the four `addTab(...)` calls with the Vietnamese labels ("1. Tải truyện", etc.).
- Cross-tab wiring:
  - `scrape_tab.project_changed → _on_project_touched`
  - `translate_tab.picker.project_selected → state.touch_project`
  - `export_tab.picker.project_selected → state.touch_project`
  - `audio_tab.picker.project_selected → state.touch_project`
- The `_on_project_touched(path)` method body (touch shared state + refresh the other three pickers).
- Launch-time "reopen last project" block (lines 71–76) — but only the *first* workspace does the reopen; see §6.
- `shutdown()` — iterate the inner tabs and call `tab.shutdown()` (moved from `MainWindow.closeEvent`'s loop).

### Stays in `MainWindow` (app-global)

- The single shared `AppState` instance (`self.state`), created once and **injected** into every `Workspace`.
- Settings button/menu (`_open_settings`, `settings_action`, the "&App" menu) and Library action (`_open_library`).
- `window_geometry` save/restore.
- Dock-badge `changeEvent` (`clear_dock_badge` on activation) — stays app-global (see §5).
- The outer `QTabWidget` of workspaces, New/Close workspace actions, corner button.
- The cross-workspace same-project guard registry (§4).

### `Workspace` public contract

```python
class Workspace(QWidget):
    title_changed = Signal(str)   # loaded novel title → outer tab label
    project_opened = Signal(str)  # project path → host registers it (guard)
    project_closed = Signal(str)  # optional, for guard cleanup on tab close

    def __init__(self, config: AppConfig, state: AppState, parent=None): ...
    def shutdown(self) -> None: ...          # shuts down its four tabs
    def has_running_workers(self) -> bool: ...  # for close confirmation
    def current_project_path(self) -> str: ...  # for the guard on close
```

- Constructor: builds the inner `QTabWidget` + four tabs (config passed to each, unchanged), wires the four connections above using the *injected* `state`.
- **Do NOT** call `refresh_recent`/`refresh_projects` in the constructor by default — make reopen opt-in via a param or a separate method (`open_last_project()`), so only workspace #1 reopens on launch and new workspaces start empty. Suggested signature: `Workspace(config, state, reopen_last=False)`; when `True`, run the current lines 71–76 logic. Alternatively expose `def reopen_last_project(self): ...` and let `MainWindow` call it on the first workspace only. Prefer the explicit method — clearer and testable.

### Title reporting (outer tab label)

- `scrape_tab.project_changed(path)` fires on load/scan (lines 154, 191, 370). In `_on_project_touched`, after touching state, derive a display title and emit `title_changed`.
- Title source: at emit time `scrape_tab.project` is set, so `scrape_tab.project.meta.title` is available. To avoid reaching into the tab's internals, add a tiny read-only accessor on `ScrapeTab` (e.g. `current_title() -> str` returning `self.project.meta.title if self.project else ""`) — this is the one small tab tweak. Fallback if empty: keep the current outer label. Truncate long titles for the tab (e.g. 24 chars + "…").

## 2. MainWindow becomes the workspace host

New structure:

```python
class MainWindow(QMainWindow):
    def __init__(self, config):
        ...
        self.state = AppState()                     # single shared instance
        self.workspaces = QTabWidget()
        self.workspaces.setTabsClosable(True)
        self.workspaces.setMovable(True)
        self.workspaces.tabCloseRequested.connect(self._close_workspace)
        self.setCentralWidget(self.workspaces)
        self._open_paths: dict[str, Workspace] = {}  # guard registry
        self._add_workspace(reopen_last=True)        # workspace #1, current behaviour
```

- `_add_workspace(reopen_last=False) -> Workspace`:
  - Construct `Workspace(self.config, self.state)`.
  - Connect `ws.title_changed → lambda t: self._set_ws_title(ws, t)`.
  - Connect `ws.project_opened → self._register_project` and `ws.project_closed → self._unregister_project`.
  - `idx = self.workspaces.addTab(ws, self._default_ws_label())`, `setCurrentIndex(idx)`.
  - If `reopen_last`, call `ws.reopen_last_project()`.
- Default outer label: "Truyện 1", "Truyện 2", … (a monotonically increasing counter so labels stay stable when middle tabs close). Once a novel loads, `title_changed` replaces it with the truncated novel title.

## 3. New / Close workspace UX + corner-widget resolution

### Corner-widget conflict

The `TopRightCorner` currently holds the Settings button. Resolve by moving Settings to the **top-left corner** and putting the **New Workspace "+" button in the top-right corner** of the *outer* `workspaces` tab bar. Rationale: "+" for a new tab conventionally sits at the tab-strip end (right); Settings is app-global and fine on the left. The Settings QAction in the "&App" menu (PreferencesRole) stays as-is, so macOS users still get it under the app menu.

```python
self.new_button = QPushButton("＋")
self.new_button.setToolTip("Mở truyện mới (Cmd/Ctrl+T)")
self.new_button.clicked.connect(lambda: self._add_workspace())
self.workspaces.setCornerWidget(self.new_button, Qt.Corner.TopRightCorner)
self.workspaces.setCornerWidget(self.settings_button, Qt.Corner.TopLeftCorner)
```

### Keyboard shortcuts (QActions added to MainWindow, or the menu)

- New Workspace: `QAction` with `QKeySequence.StandardKey.AddTab` (Cmd/Ctrl+T) → `_add_workspace()`. Add to the "&App" menu too for discoverability.
- Close Workspace: `QAction` with `QKeySequence.StandardKey.Close` (Cmd/Ctrl+W) → `_close_workspace(self.workspaces.currentIndex())`.

Note: on macOS Cmd+W traditionally closes the window. Here it closes the current workspace tab; when only one workspace remains, closing it should follow the "last workspace" rule below (so Cmd+W on the last tab effectively quits — matching user expectation).

### Close semantics (`_close_workspace(index)`)

1. `ws = self.workspaces.widget(index)`.
2. If `ws.has_running_workers()`: show a `QMessageBox.question` ("Truyện này đang chạy (tải/dịch/audio). Đóng và huỷ?"). If user cancels, return without closing.
3. Call `ws.shutdown()` (cancels + waits **only this workspace's** workers).
4. `self._unregister_project(ws.current_project_path())`.
5. `self.workspaces.removeTab(index)` and `ws.deleteLater()`.
6. **Last workspace rule:** if `self.workspaces.count() == 0` after removal, create a fresh empty workspace (`self._add_workspace()`) rather than leaving an empty window or quitting. Justification: keeps the window in a usable state, mirrors browser "last tab closed → new blank tab" behaviour, and avoids an ambiguous empty `QMainWindow`. (If the user wants to quit, the OS window close / Cmd+Q path still runs `closeEvent` → shuts all down.) Document this choice; a "quit on last close" variant is a one-line alternative if preferred later.

### MainWindow.closeEvent (rewritten)

```python
def closeEvent(self, event):
    self.config.window_geometry = self.saveGeometry()
    for i in range(self.workspaces.count()):
        ws = self.workspaces.widget(i)
        if hasattr(ws, "shutdown"):
            ws.shutdown()
    super().closeEvent(event)
```

`has_running_workers()` can also gate a global "workers running, quit anyway?" confirm on window close — optional, recommend keeping it simple and just shutting down (current behaviour never confirmed).

## 4. Same-project-open guard

Mechanism: MainWindow keeps `self._open_paths: dict[str, Workspace]` mapping an open project path → the owning workspace.

Hooks (grounded in real signals):
- `Workspace.project_opened(path)` is emitted whenever the workspace's ScrapeTab fires `project_changed` (load at 154, scan at 191, resume at 370) **or** any of the three pickers' `project_selected` fires. In practice, route everything through the workspace's existing `_on_project_touched(path)` (which already runs for scrape) plus add the same registration when a picker selects a project. Simplest: `Workspace` emits `project_opened(path)` from a single internal handler that all four sources funnel into.

Guard flow (`MainWindow._register_project(ws, path)`):
1. Look up `owner = self._open_paths.get(path)`.
2. If `owner` is `None` or `owner is ws`: record `self._open_paths[path] = ws`; also drop any stale prior path owned by `ws` (a workspace only holds one project at a time). Update tab label.
3. If `owner` is a *different* workspace: this is a conflict.

Conflict handling (pragmatic "warn + focus existing", the recommended minimal guard):
- Show `QMessageBox.warning`: "Truyện này đang mở ở một tab khác. Chuyển sang tab đó." then `self.workspaces.setCurrentWidget(owner)`.
- Tell the offending workspace to revert: the cleanest revert depends on how far the open got. Because full rollback of an in-progress load is invasive, the recommended pragmatic scope is:
  - **Refuse before heavy work where cheap:** for *picker selection* (translate/export/audio) the guard can veto before the tab loads by checking the host in the `project_selected` handler — but that requires the pickers to consult the host, which crosses the Workspace/host boundary.
  - **Realistic minimal implementation:** allow the load to complete but immediately surface the warning and focus the existing owner; leave the duplicate open in a read-mostly state. Because SQLite `NovelProject` opens are the actual corruption risk, register the path at open time and, on conflict, prefer *not* registering the duplicate (so the original stays the owner) and warn. Document residual risk: two tabs can still both have the project open if the user dismisses the warning and forces work; a hard lock (e.g. a lockfile / SQLite `BEGIN EXCLUSIVE` probe) is the robust fix and is noted as a follow-up.

Recommended decision for this iteration: **warn-and-focus** on duplicate, registry-based, no forced rollback. Note the residual footgun in the plan and flag a lockfile-based hard guard as a follow-up (§ Risks). This keeps the change contained to `MainWindow` + one signal on `Workspace`, with no invasive changes to the tabs or storage.

Cleanup: on workspace close, `_unregister_project` removes its path from the registry (step 4 of close). When a workspace switches projects, unregister the old path before registering the new one.

## 5. Dock badge + shared AppState

### Shared AppState

- One `AppState()` created in `MainWindow.__init__`, injected into every `Workspace`. This fixes the concurrent-clobber risk (multiple `AppState()` instances writing the same JSON). `touch_project` still saves on each call; that's fine for a single instance.
- Cross-workspace recent-list refresh: keep it simple — a workspace refreshes its own pickers when it touches a project (existing behaviour). Other workspaces do **not** eagerly refresh. Recommended lazy refresh: refresh a workspace's pickers when its *inner* tab or its *outer* workspace tab becomes active. Minimal viable version: do nothing extra — pickers already refresh on their own actions and on next explicit refresh; the recent list is a convenience, not correctness-critical. Recommend documenting "recent list in a background workspace may be one project stale until you interact with it" and only add an on-activate `refresh` if it's trivial (connect `workspaces.currentChanged → ws.refresh_pickers()` where `refresh_pickers` calls the three `refresh_projects()` + `refresh_recent()` with no select). This one connection is cheap; include it.

### Dock badge

- `notify.set_dock_badge/clear_dock_badge/request_attention` are app-global and operate on `QApplication.instance()`. ScrapeTab calls `set_dock_badge(1)` / `clear_dock_badge()` / `request_attention(self.window())` directly (lines 216, 267–268, 279–280, 303, 312–313, 325–326).
- Decision: **keep the badge app-global with "count = number of workspaces needing attention"**, but only if cheap. Two viable options:
  - (A) Minimal, recommended: leave `notify.py` unchanged and accept "any workspace hitting the daily limit sets the badge; app activation clears it." The badge means "at least one flow needs you." Justification: the badge is a coarse attention cue, `changeEvent → clear_dock_badge()` already clears on focus, and `request_attention` bouncing is inherently app-global. Zero code change, no races that matter (both workspaces setting `1` is idempotent-ish; a stale badge is cleared on next focus).
  - (B) Accurate counter: add a small counter in `MainWindow` (`_attention_count`) that workspaces increment/decrement via a signal, and call `set_dock_badge(count)`. This requires ScrapeTab to stop calling `set_dock_badge` directly and instead emit an "attention" signal up to the workspace/host. That's a broader change to the tab.
- Recommend **(A)** for this iteration; note (B) as an enhancement. If the reviewer wants accuracy, the seam is: ScrapeTab emits `needs_attention`/`attention_cleared` signals instead of calling notify directly; Workspace forwards; MainWindow keeps the counter.

## 6. Session persistence

Out of scope for this iteration (recommended). Keep launch behaviour minimal: **one workspace on launch, reopen last project as today** (workspace #1 only, via `reopen_last_project()`). Multi-workspace session restore (persisting the set of open project paths and reopening them) is a clean follow-up; note it explicitly. No behaviour regression: users still land on their last novel.

## Testing

Headless (offscreen `qapp` fixture, `QT_QPA_PLATFORM=offscreen`):

- `test_main_window_starts_with_one_workspace`: construct `MainWindow(AppConfig())`; assert `workspaces.count() == 1` and the widget is a `Workspace`.
- `test_new_workspace_is_independent`: call `_add_workspace()`; assert `count() == 2`, and the two workspaces' `scrape_tab` objects (and their worker attributes) are distinct instances (`ws1.scrape_tab is not ws2.scrape_tab`).
- `test_shared_appstate`: assert every workspace's injected `state` is the *same* object (`ws1.state is main.state is ws2.state`).
- `test_close_workspace_shuts_down_only_itself`: monkeypatch/spy `shutdown` on both workspaces (or on their tabs); `_close_workspace(1)`; assert the closed workspace's `shutdown` ran and the other's did not.
- `test_close_last_workspace_refills`: close down to zero; assert `count() == 1` again (fresh empty workspace).
- `test_same_project_guard`: register a project path in ws1 (call `_register_project(ws1, path)`), then attempt to register the same path from ws2; assert the registry still maps to ws1 and (patch `QMessageBox.warning`) the warning fired and current widget switched to ws1.
- `test_worker_running_confirm`: stub `has_running_workers → True`, patch `QMessageBox.question` to return "No"; assert `_close_workspace` did not remove the tab.

Not practically headless (manual): actual parallel downloads, real QThread worker lifecycles under load, dock-badge rendering, real daily-limit handling per workspace.

## Manual verification

1. Launch → one workspace, last project reopened (unchanged behaviour).
2. Cmd/Ctrl+T (or "+") → second workspace, empty.
3. In ws1 scan+download novel X; in ws2 scan+download novel Y simultaneously. Confirm both progress bars advance independently and neither tab's status bleeds into the other.
4. Trigger the daily limit in ws1; confirm the dock badge + attention bounce, and that ws2 keeps running.
5. Open novel X in ws2 while it's open in ws1 → warn + focus ws1.
6. Close ws1 while it's downloading → confirm dialog → only ws1's workers cancel; ws2 unaffected.
7. Close the last remaining workspace → a fresh empty workspace appears (window not empty).
8. Quit the window (Cmd+Q / window close) → all workspaces shut down cleanly; relaunch restores geometry + last project.

## Risks / open questions

- **Same-project guard is warn-only.** A determined user can dismiss the warning and force two workspaces onto one SQLite project → lock/corruption risk. Follow-up: a hard guard via a per-project lockfile or `BEGIN EXCLUSIVE` probe on open. Confirm the reviewer accepts warn-and-focus for v1.
- **Guard registration timing.** Because `project_changed`/`project_selected` fire *after* the project is opened, the guard warns post-open rather than pre-empting the SQLite open. Acceptable for v1; a pre-open veto needs the tabs/pickers to consult the host, which is more invasive.
- **Dock badge coarseness (option A).** "Any workspace needs attention" rather than an accurate count; cleared on app focus. Confirm acceptable vs. the counter (option B) which requires ScrapeTab to emit signals instead of calling `notify` directly.
- **Cross-workspace recent-list staleness.** Background workspaces refresh pickers only on activation; a just-touched project may appear one step stale until the tab is focused. Acceptable.
- **Session restore deferred.** Only one workspace reopens on launch; multi-workspace restore is a follow-up.
- **Settings live-apply.** Settings writes shared `AppConfig` (QSettings). Each tab reads `self.config` on demand (e.g. `library_dir`, `request_delay`, cookies at worker-creation time), so changes apply to all workspaces on next action. Verify no tab caches config values at construction that would go stale across workspaces.
