# 007 — Multiple windows

## Request

> Can you add multiple windows, so I could crawl 2 or more different flows at the
> same time?
>
> Clarified: "multiple windows like multiple big tabs for each" — i.e. an outer bar
> of top-level "big tabs" (workspaces), each big tab an independent crawl flow.

Let the user run 2+ independent scrape/download (and translate/TTS) flows in parallel
inside **one app window**, via an outer "workspace" tab bar. Each workspace is a full,
independent instance of the current tab stack — its own novel/project, its own
scrape/translate/TTS sub-tabs, and its own workers — e.g. workspace A downloading
novel X while workspace B downloads novel Y.

**Decided model:** outer `QTabWidget` of workspaces (NOT separate OS-level windows,
NOT tabs-within-a-single-flow). Add/close/rename workspace tabs; a "+" / New Workspace
action.

## Current state (verified)

- `src/noveltrans/app.py` creates a single `MainWindow(AppConfig())`, `.show()`s it,
  and runs `app.exec()`.
- `MainWindow` (`gui/main_window.py`) is a `QMainWindow` holding the tabs; its
  `closeEvent` calls `tab.shutdown()` on each tab to cancel running workers.
- Workers (scan/download/unlock/translate/TTS) are `QThread`s owned by their tab.
- Config is `AppConfig` over `QSettings` (process-wide, shared).

## Scope (to confirm during /plan)

- Refactor the current tab stack (scrape/translate/TTS) into a reusable "workspace"
  widget that owns its own project + workers, then host N of them in an outer
  `QTabWidget` inside `MainWindow`.
- A New Workspace action (`Cmd/Ctrl+T`, or a "+" corner button) and close-tab
  (`Cmd/Ctrl+W`) support; ideally rename tabs (default to the loaded novel title).
- Each workspace shuts down only its own workers; closing a workspace with running
  workers should confirm/cancel cleanly. `MainWindow.closeEvent` shuts down all
  workspaces.
- Start with one workspace on launch (current behaviour preserved).

## Open questions / risks

- **Extract the cross-tab wiring:** `MainWindow` currently owns the four tabs AND
  their wiring — `AppState`, `scrape.project_changed → _on_project_touched`, the
  pickers' `project_selected → state.touch_project`, and launch-time
  "reopen last project". All of this must move into the new `Workspace` widget so each
  workspace is self-contained; `MainWindow` keeps only app-global bits (Settings
  button/menu, Library, `window_geometry`, dock-badge `changeEvent`).
- **Same project opened in two workspaces:** storage is SQLite per project — two
  workspaces writing the same project path could lock/corrupt. Need a guard (detect
  the path is already open in another workspace → warn/prevent) or at least document
  the footgun.
- **`AppState` last-project:** with N workspaces, "reopen last project on launch"
  applies to the first workspace only; touching a project in any workspace updates the
  shared recent list. Decide whether recent-project lists refresh across workspaces.
- **App-global side effects:** Dock badge and `request_attention` (gui/notify) are
  application-level; two workspaces setting/clearing the badge will race. Decide
  whether the badge is per-workspace-meaningful or app-global.
- **Shared config:** Settings dialog writes `AppConfig` (shared QSettings); a change
  should apply to all workspaces — check for stale in-memory reads.
- **Close semantics:** closing a workspace tab with running workers must confirm and
  `shutdown()` only that workspace's workers; closing the last workspace shouldn't
  leave an empty window (reopen a fresh one, or close the app?).

_Investigate the Workspace extraction + QThread ownership during /plan before
implementing._
