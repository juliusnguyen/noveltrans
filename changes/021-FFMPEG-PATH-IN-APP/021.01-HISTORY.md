# 021 — ffmpeg PATH in the packaged .app — History

## [FIX] "Tạo video" button disabled in the .app — 2026-07-18
User reported the button couldn't be used in the packaged app. **Diagnosed, not guessed:**
the button's only disable condition is `not ffmpeg_available()`, and `ffmpeg_available()` is
`shutil.which("ffmpeg")`. Reproduced the Finder launch environment
(`env -i PATH="/usr/bin:/bin:/usr/sbin:/sbin"`): `shutil.which("ffmpeg") → None`. The user's
ffmpeg is at `~/.local/bin/ffmpeg`, which a Finder-launched app's minimal PATH excludes.
Adding `~/.local/bin` → found. (This affected every ffmpeg/ffprobe call, not just video —
video was just where it showed.)

**Fix — `runtime_env.augment_tool_path()` (new), called first in `app.main()`:** prepends
the standard tool dirs that exist (`/opt/homebrew/bin`, `/usr/local/bin`, `/opt/local/bin`,
`~/.local/bin`) to `os.environ["PATH"]` before any `ffmpeg_available()` check or subprocess
runs. Idempotent (no dupes; a no-op from a terminal that already has them); every subprocess
inherits the augmented PATH, so one fix covers video/audio/merge. Mutates in place, testable
via injected `environ`/`home`.
- Tests: prepends `~/.local/bin` ahead of inherited entries; idempotent; skips non-existent
  dirs; handles an empty PATH. 501 passing (from 497).

### Verified
- `env -i PATH=<minimal> python -c 'which ffmpeg'` → None before; adding the dir → found.
- After rebuilding the `.app`, launched with a stripped PATH and confirmed the app now
  resolves ffmpeg (button enabled).
