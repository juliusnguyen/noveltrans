# 015.02 — REVISED Plan: 69shuba.com source support

> Supersedes the architecture in `015.01-INITIAL-PLAN.md`. Steps 1–2 of that plan are
> DONE and committed (`63af59b`, `230ed09`); Step 2 empirically killed its core design.
> Read `015.01-HISTORY.md` for the findings. Start here at Step 3.

## 0. Summary

Steps 1–2 are done and committed. Step 2 killed the hybrid design empirically: 69shuba's Cloudflare gate passes real Chrome **silently on TLS/browser fingerprint** and issues **no `cf_clearance` at all** (only `zh_choose`). There is no token to harvest, so `requests` can never be made to work here at any header-copying effort.

**The architecture is now always-Playwright.** `Shuba69Adapter._get_html()` drives a browser for every page. This flips the plan's centre of gravity: the `base.py`/`HttpClient` work is dropped entirely, and the new hard problem is **browser lifecycle and per-navigation cost across 199 chapters** — a problem the initial plan never costed because it assumed bulk fetching over `requests`.

Because the site's HTML is now captured and read, every selector below is **verified against the real fixtures**, not guessed. The remaining unknowns are narrow and named in §7.

## 1. Revised architecture

```
ScanWorker.run() / DownloadWorker.run()      (QThread — never the GUI thread)
  └─ adapter = adapter_for_url(url, client)   → Shuba69Adapter
     adapter.on_status = <worker callback>    → Vietnamese status strings
     try:  … fetch_metadata / fetch_chapter_list / fetch_chapter ×199 …
     finally: adapter.close()                 → tears the browser down, always

Shuba69Adapter
  ├─ _session: BrowserSession | None          (lazy — built on first fetch)
  ├─ _get_html(url) -> str                    THE SEAM: session.get_html(url)
  └─ parse_metadata / parse_chapter_list / parse_chapter   ← module-level PURE fns
                                                 (markup: str) -> …  no browser, no I/O

cf_browser.BrowserSession                     (new module)
  ├─ __enter__ / close()                      one persistent context for a whole batch
  ├─ one reused `page`, one `page.route()` resource blocker
  ├─ _throttle()                              own politeness delay (HttpClient's is bypassed)
  └─ get_html(url) -> str                     goto → wait → looks_like_challenge? → content()
```

### Why this shape

- **`_get_html` was always designed as the only seam** (initial plan §1). The pivot lands entirely inside it. Parsing, fixtures and tests are unaffected — which is exactly why Step 2 was sequenced first.
- **Fetch/parse split is now mandatory, not nice-to-have.** With `requests`, `responses` could fake the transport and the adapter stayed testable end-to-end. With Playwright there is no equivalent — so parsing must be pure functions taking `markup: str`, and the browser must sit behind a seam thin enough to fake in three lines. Tests then never launch a browser.
- **The adapter's lifetime is already correct for owning a browser.** `DownloadWorker.run()` (`workers.py:766-811`) constructs the adapter **once** and loops all pending chapters against it; `ScanWorker.run()` (`workers.py:51-77`) does two fetches against one adapter. So "one context per adapter, lazily created" == "one context per batch" with no new ownership machinery. What's missing is teardown — `SiteAdapter` has no `close()`, so the workers currently just drop the adapter on the floor. That's the one structural addition needed (§3, Step 6).

## 3. Ordered implementation steps

### Step 3 — `src/noveltrans/cf_browser.py` (NEW) — the browser session

The load-bearing new module. Sibling of `discord_unlock.py`, built on `browser.py` (Step 1).

**Docstring must be candid** — following `discord_unlock.py:22-23`'s precedent verbatim in spirit: state plainly that 69shuba's operator deliberately deployed a Cloudflare challenge, that this module drives a real browser past it, that the polite delay must stay on, and that this is intended for personal reading volume. The repo already writes this kind of thing down rather than leaving it implicit; do the same here. (015.01 §5 R6 — still the user's call to make, but it belongs in writing.)

```python
class BrowserSession:
    """One persistent Chrome context, reused for an entire scan/download batch."""
    def __init__(self, profile_dir: Path, *, headless: bool = False,
                 delay_seconds: float = 1.5, timeout_ms: int = 60_000): ...
    def get_html(self, url: str) -> str: ...
    def close(self) -> None: ...        # never raises
    def __enter__/__exit__               # so scripts/tests can use `with`
```

- `profile_dir()` → `DEFAULT_LIBRARY_DIR / ".cf-profile"` — **separate from `.discord-profile`**, as `browser.py`'s docstring already asserts.
- **Lazy launch inside `get_html`**, not `__init__`: constructing an adapter must never start Chrome (`adapter_for_url` runs in tests and on cancelled scans).
- **One page for the whole batch**, `context.pages[0] or context.new_page()`. Rationale: a new page per chapter re-pays context/route setup ~199×; there is no isolation benefit (same origin, same cookies, and there's no cookie state to protect). Risk is JS heap growth across 199 navigations — negligible at this scale on a static novel site; if a multi-thousand-chapter book ever leaks, recycle the page every N navigations. Don't build that now.
- **Resource blocking** via a single `page.route("**/*", handler)` registered once:
  ```python
  _BLOCKED_TYPES = {"image", "font", "media", "stylesheet"}
  # NEVER block "document", "script" or "xhr" — Cloudflare's check is JS-driven.
  ```
  Expected win: the captured chapter page carries 5 `<script>` blocks, an `<ins>` ad slot and three ad divs (`#ad-first-slot-pc`, `#ad-second-slot-pc`, `#ad-third-slot-pc`) plus a cover image and site CSS. Blocking images/fonts/media/CSS drops the great majority of bytes and most of the request count; a text-only navigation should land near ~0.5–1s versus ~2–3s fully loaded. Over 199 chapters that's roughly 3–6 minutes saved — meaningful, but note it is **second-order next to the politeness delay** (199 × 1.5s ≈ 5 min floor regardless). Keep `_BLOCKED_TYPES` a module constant: if 403s ever appear, flipping it to `set()` is the first diagnostic. (Theoretical risk that CF notices a page fetching no subresources; the gate is decided at connection on TLS fingerprint, so this is unlikely — but it's the cheapest thing to rule out.)
- **`wait_until="domcontentloaded"`**, then `page.wait_for_selector` on a content marker. Do **not** use `networkidle` — with ad scripts blocked/pending it may never settle. (`discord_unlock.py:216` already wraps `networkidle` in a swallow-the-timeout guard for the same reason.)
- **`_throttle()`** — copy the shape of `HttpClient._throttle` (`base.py:48-51`): `time.monotonic()`, sleep the remainder. The `HttpClient` throttle is fully bypassed on this path, so **without this the adapter would hammer the site with zero delay** — the single most dangerous omission in the whole feature. Seed `delay_seconds` from `client.delay_seconds`. Note the *effective* rate is delay + navigation time, i.e. naturally politer than `requests`.
- **`looks_like_challenge(markup) -> bool`** — module-level, pure, no Playwright import. Reuse the marker tuple already proven in `scripts/capture_69shuba.py:47-53` (`challenges.cloudflare.com`, `cf-browser-verification`, `__cf_chl`, `Just a moment`, `Enable JavaScript and cookies to continue`). `get_html` checks the returned content and raises rather than handing an interstitial to the parser (which would otherwise fail with a misleading "layout may have changed").
- **Module-level `threading.Lock`** around launch. `DownloadWorker.run()` is sequential today, but 014 shipped a `ThreadPoolExecutor` in `AudioWorker` (`workers.py:516`) and the parallel direction is live. Two threads launching Chromium on the same profile dir corrupts it. ~3 lines; add it now.
- `BrowserUnavailableError` → caught and re-raised as a Vietnamese `ScrapeError` by the adapter, mirroring `discord_unlock._require_playwright()` (`discord_unlock.py:81-90`).

**Sub-step 3a — settle headed vs headless with evidence, not argument.** See §5; run `scripts/capture_69shuba.py` with `headless=True` once. It's a ten-minute experiment that converts the plan's biggest UX question into a fact. Do this *before* wiring the default.

**Sub-step 3b — capture `tests/fixtures/69shuba/challenge.html`.** The 5.6KB interstitial was observed but never saved. `ixdzs/challenge.html` is the precedent. A plain `requests.get` on any 69shuba URL returns it — trivial to add to the capture script.

**The title-dup trap — designed and justified.**

Verified:
| source | value |
|---|---|
| TOC / `h1.hide720` (ch.1) | `1.第1章 这是什么地狱开局？` |
| body line 0 (ch.1) | `第1章 这是什么地狱开局？` |
| TOC (ch.2) | `第2章 剧情开始，第一女主出现` |

The separator is ASCII `.` (U+002E) and the space is ASCII (U+0020) in **both** — so no whitespace normalisation is needed. The site is simply inconsistent about a leading `N.` index prefix (present on ch.1, ch.198, ch.199; absent on ch.2, ch.3). `xbanxia.py:84`'s `lines[0] == ref.title` would silently fail to strip on exactly the prefixed chapters.

Design — one normaliser, applied on both sides and at TOC-parse time:
```python
_INDEX_PREFIX_RE = re.compile(r"^\s*\d+\s*[.．、]\s*")
def _clean_title(text: str) -> str:
    return _INDEX_PREFIX_RE.sub("", text).strip()
```
1. `parse_chapter_list` stores `_clean_title(a.get_text(strip=True))` → titles become uniform (`第1章 …`, `第2章 …`). This is also a *user-facing* win: the `N.` prefix is pure duplication of the `第N章` already in the title, and it would otherwise leak into the EPUB TOC and get read aloud by TTS. Feature 011 (`af096b9`, stripping medoctruyen status badges from titles) is the precedent for normalising site noise out of titles.
2. `parse_chapter` strips line 0 iff `_clean_title(lines[0]) == _clean_title(ref.title)` — normalising **both** sides, because `ref.title` comes from the project DB and may predate this normalisation.

**Why exact-equality-after-normalisation and not `endswith`/fuzzy** (015.01's suggestion): the two failure modes are wildly asymmetric. Under-stripping costs a duplicated heading line — cosmetic. Over-stripping silently deletes a real first paragraph — data loss the user won't notice until they read it. `ref.title.endswith(line)` would happily eat a short opening line of dialogue. So: remove exactly one well-defined prefix, then demand equality, and **never** remove more than one line. Bias hard to under-strip.

**Testable seam, exactly:**
```python

def parse_chapter_list(markup: str, base_url: str) -> list[ChapterRef]
def parse_chapter(markup: str, title: str, url: str) -> str
```
The adapter methods become three lines each: derive URL → `self._get_html(...)` → `parse_*(markup, …)`. `_get_html` is the *only* method that touches the browser.

## 5. The headed/headless recommendation

**First, kill an option 015.01 implied:** "solve headed once, then continue headless/fast" is **impossible here**. There is no token — no `cf_clearance`, nothing to carry forward. The live browser session *is* the clearance, for all 199 navigations. So the real choice is binary: headed for the whole batch, or headless for the whole batch.

The tension is sharp and worth stating plainly:
- **For headed:** the gate here is *purely* fingerprint. Headless Chrome advertises `HeadlessChrome` in its UA and differs in measurable ways; `discord_unlock.py:191-193` already documents headless "invites a mid-run re-challenge" against a *weaker* check than this one. If headless is fingerprinted, **every** fetch 403s and the feature is simply broken.
- **Against headed:** a visible Chrome window flipping through 199 chapters for ~10 minutes, stealing focus on launch, is a serious UX imposition — far worse than Discord unlock's brief one-off window. And the user can close it and break the batch.

**Recommendation: headed by default, headless behind a Settings toggle — but settle it empirically in Sub-step 3a.** Run the existing capture script with `headless=True`. Two outcomes, both cheap:
- Headless passes → **flip the default to headless**, invert the toggle to *"Hiện cửa sổ trình duyệt (nếu bị chặn)"*, and the UX problem evaporates.
- Headless 403s → headed default stands, on evidence rather than analogy, and the toggle is documented as *"thử nghiệm"*.

Given the challenge cleared **passively** for real Chrome — no interstitial, no checkbox — headless has a genuine chance. That makes the experiment clearly worth ten minutes before committing the default.

Rejected: positioning the window off-screen (`--window-position=-2400,-2400`) to fake headless-with-headed-fingerprint. It's a hack, it's undocumented behaviour, and it makes a stuck browser invisible to the user.

**Parallelism — a live conflict, call it out.** 014 (`7fd9a9a`) shipped parallel TTS workers with a `tts_workers` setting and a `ThreadPoolExecutor` (`workers.py:516`); the natural next ask is parallel *downloads*. **69shuba must never be parallelised**: N concurrent contexts is N Chromes, and hammering a CF-protected site is the fastest route to an IP block that would kill the Playwright path too — the only path this site has. 015.01 §5 R4 stands and is now more binding. Don't build opt-out machinery today; **do** write it in the adapter docstring so whoever implements parallel downloads finds the warning before the outage.

## 6. Risks

- **R1 (was: cookie won't replay) — REALISED.** Already absorbed; the architecture changed. Closed.
- **R-new-1 — headless may be fingerprinted.** Highest live risk. Retired by Sub-step 3a for the cost of one script run.
- **R-new-2 — leaked Chromium processes.** An orphan window is user-visible and infuriating. Mitigated by Step 6's `SiteAdapter.close()` + `finally`, plus `browser.close()`'s never-raises contract and `browser.py:84`'s stop-before-raise.
- **R-new-3 — Cloudflare may escalate later.** Today it's passive; a future policy change could add an interactive challenge. Headed survives it (the user clicks); headless would hang to timeout. An argument for keeping the headed path first-class even if 3a says headless works.
- **R-new-4 — resource blocking could itself be fingerprinted.** Unlikely (the gate is decided at connection), and `_BLOCKED_TYPES = set()` is a one-line diagnostic.
- **R-new-5 — throughput.** 199 chapters × (1.5s delay + ~1s navigation) ≈ 8–10 min, and it cannot be parallelised. Acceptable, and honestly not much worse than `requests` would have been given the delay floor — but the GUI must show progress so it doesn't read as a hang. `DownloadWorker.progress` already does.
- **R6 (legal/ToS) — unchanged, and now unavoidable rather than optional.** The initial plan's framing survives intact: the operator deployed this deliberately; the feature routes around it; `discord_unlock.py`'s docstring is the house precedent for saying so plainly. Keep the delay, keep the volume personal, write it down. The user's call to make, not the plan's.
- **R7 — site drift.** One snapshot, one book. `make test-live` is the early warning.
- **R-new-6 — packaging.** See §8; now blocking for the feature's usefulness in the shipped `.app`.

## 8. Packaging — restated, more acute, user's call

Before, only Discord auto-unlock needed Playwright. Now **69shuba is unusable without it.** And `make app` (`Makefile:40-45`) installs only base + pyinstaller — so in the frozen `.app`, `import playwright` fails and `browser.py:50` helpfully tells the user to run `pip install`, in a bundle that has no pip. Dead end.

1. **Add Playwright to the frozen build** — `collect_all("playwright")` in `packaging/NovelTrans.spec` (which already does exactly this for six packages, `spec:12-17`) + install it in the `make app` target. Relies on `channel="chrome"` driving the user's real Chrome, so no Chromium download. ~10MB. **Doesn't work without Chrome installed.** ⚠️ Playwright ships a **Node driver binary** under `playwright/driver/` — `collect_all` should gather it as data, but the executable bit and the `.app` codesign/quarantine story are exactly the kind of thing feature 012 (`c123e60`, bundling `sea_g2p.bin`) proved fiddly. **Needs a real test build before this is called done.**
2. **Bundle Chromium too** — ~150MB+, and `playwright install` writes to `~/Library/Caches/ms-playwright`, not the bundle. Painful.
3. **Ship without it** — a clear Vietnamese error: install Chrome and run from source for 69shuba.

**Lean (1)**, but it changes what "the app works" means for a user without Chrome, and its risk is concentrated in an untested build step. **This is the user's decision.** Steps 3–9 are green regardless — the source-run path works today.
