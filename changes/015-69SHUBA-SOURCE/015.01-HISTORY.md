## [RESEARCH] Cloudflare challenge blocks the plain-requests approach — 2026-07-16
Probed the target site before planning. `https://www.69shuba.com/book/59024/` returns
**HTTP 403** with `cf-mitigated: challenge` and a CSP referencing
`challenges.cloudflare.com` — a Cloudflare **managed challenge**, not the trivial JS
token that `ixdzs` solves inline.

Confirmed the block is site-wide, not host-specific — all 403:
- `www.69shuba.com`, `69shuba.com`, `www.69shu.com`, `www.69shuba.cx`
- (`m.69shuba.com` does not resolve)

Consequence: a `requests`-based adapter in the shape of `xbanxia`/`ixdzs` cannot work
here — every fetch returns the interstitial rather than the book page. The real page
HTML remains **unseen**, so no selectors can be written until a browser session
captures it.

**Decision (user):** solve the challenge with a **headless browser (Playwright)**.
Alternatives raised and declined: pasting a `cf_clearance` cookie from the user's own
browser (the `medoctruyen` precedent — fragile, UA/IP-bound, expires mid-run);
TLS-fingerprint impersonation via `curl_cffi` (declined — its purpose is defeating bot
detection); dropping 69shuba for a plain-HTML source.

Materially lowers the cost of that choice: **Playwright is already a dependency** —
`pyproject.toml` carries `playwright>=1.40` under the `discord` extra, and
`src/noveltrans/discord_unlock.py` already establishes the house pattern (lazy import
with install instructions, `launch_persistent_context` on a dedicated profile dir,
real-Chrome-then-bundled-Chromium channel fallback, `--disable-blink-features=
AutomationControlled`). The new adapter should follow that module rather than invent
its own browser handling.

## [STEP 1] Extract shared browser module — 2026-07-16
Plan §2 Step 1. Pure refactor of working code (feature 004), no behaviour change.

**Added `src/noveltrans/browser.py`** — the launch machinery formerly private to
`discord_unlock.py`, now shared with the upcoming Cloudflare solver:
- `BrowserUnavailableError` — generic, English. This module is infrastructure; the
  app speaks Vietnamese, so callers catch and re-raise their own strings.
- `require_playwright()` — lazy import + install hint (`noveltrans[browser]`).
- `launch_persistent_context(sync_playwright, profile_dir, *, headless)` — real Chrome
  then bundled Chromium, `--disable-blink-features=AutomationControlled`,
  `ignore_default_args=["--enable-automation"]`. Stops Playwright before raising so no
  process leaks. Takes `sync_playwright` as an argument so tests can inject a fake;
  takes `profile_dir` as an argument so each caller keeps its own profile.
- `close(context, playwright)` — never-raises teardown.

Rationale for extracting rather than duplicating: these are anti-bot-detection args
whose entire value is being correct and consistent. Two copies drifting apart would
mean one caller silently losing the disguise.

**Rewired `src/noveltrans/discord_unlock.py`** — `_require_playwright()` and
`_launch_context()` are now thin shims that delegate and translate
`BrowserUnavailableError` back into the Vietnamese `DiscordUnlockError`. Removed the
moved `_BROWSER_CHANNELS`, `_LAUNCH_ARGS` and `_close`; `_close` is now imported from
`browser`. Discord-specific DOM selectors (`_PICKER_OPTION_SEL` etc.) stayed put.

**`tests/test_discord_unlock.py` needed no changes** — the plan flagged it as likely
to break since it imports `_launch_context` and `_PICKER_OPTION_SEL` directly. Keeping
`_launch_context` as a shim with the same signature preserved both, so the existing
tests still exercise the new code path through the shim (incl. the Chrome-preferred,
bundled-fallback and no-browser-at-all cases, and the `fake.stopped` no-leak assertion
which now covers `browser.py`'s stop-before-raise).

Verified: `256 passed, 5 deselected`; `ruff check src tests` clean. Green in isolation
before proceeding, per the plan's instruction.

## [STEP 2] Capture fixtures — R1 FAILED, hybrid design is dead — 2026-07-16
Plan §2 Step 2 (the blocking step). Added `scripts/capture_69shuba.py`. Installed
`playwright>=1.40` into the venv (`uv pip install`); real Chrome is present at
`/Applications/Google Chrome.app`, so the `channel="chrome"` path is used and no
bundled-Chromium download was needed.

### R1 failed — and worse than the plan predicted
The browser cleared the challenge and reached the real page. Replaying its
User-Agent + cookies over plain `requests` still returned **403 `cf-mitigated:
challenge`**.

The cause is not a cookie that failed to transfer — **no `cf_clearance` cookie was
issued at all**. The only cookie set was `zh_choose` (a language preference). Real
Chrome was passed *silently on fingerprint*, without an interstitial and without a
token. So there is nothing to replay: Cloudflare here gates on TLS/browser
fingerprint, which `requests` cannot reproduce at any header-copying effort. The
hybrid is not "fragile", it is **impossible**.

Also confirmed the plan's UA point: the real Chrome UA is `Chrome/150.0.0.0`, against
the `Chrome/126` hardcoded in `base.py:14`.

**Consequence — the architecture flips to always-Playwright** (`_get_html` drives the
browser), exactly the fallback plan §1 named. Because `_get_html` was designed as the
only seam, parsing/tests/fixtures survive the pivot. Knock-on effects:
- **Plan §2 Step 4 (`base.py` changes) is now largely unnecessary.** 69shuba never
  touches `HttpClient`, so the UA override (4a), the `requests`-side challenge check
  (4b) and the forced GBK encoding (4c) have no user. Chrome owns decoding, so the
  encoding question evaporates. Keep only `is_challenge_html(markup)` for spotting an
  interstitial in `page.content()`; drop `is_challenge_response(response)`.
- **Plan §3's `load_fixture_bytes` is unnecessary.** Fixtures are `page.content()` —
  already-decoded UTF-8, exactly what the adapter receives — so the existing
  `load_fixture` is correct and `responses` isn't involved at all. Tests should feed
  saved markup to pure parsing helpers, which keeps them browser-free. This argues for
  splitting fetch from parse in the adapter.
- **New concern the plan didn't cost:** 199 chapters × one navigation each. Needs one
  persistent browser for the batch plus resource-blocking (images/fonts/media), not a
  launch per chapter. Reinforces plan §5 R4: do not parallelize this site.

### Fixtures captured (`tests/fixtures/69shuba/`)
69shuba splits across three pages, where other sites use one — the plan assumed one:
| fixture | url | what |
|---|---|---|
| `novel.html` | `/book/59024.htm` | info page — title/author/cover/description |
| `toc.html` | `/book/59024/` | chapter list (199 links) |
| `chapter.html` | `/txt/59024/38369377` | one chapter body |

A first attempt captured a login page as `chapter.html`: the link matcher tested
`"/article/" in href` and hit `/modules/article/bookcase.php` ("add to bookcase"),
which bounces to `用户登录`. Fixed to require `/txt/\d+/\d+`; bad fixture deleted and
re-captured.

### Plan §6 checklist — ANSWERED
**Encoding (1-3):** moot. Playwright hands over decoded text; Chrome does the decoding.
Drop Step 4c. (The site's raw charset stays unknown — `requests` never saw a real page.)

**URLs (4-7):**
- Info page is `/book/<id>.htm`; `/book/<id>/` is the **TOC**, a separate page. The
  adapter must fetch both, and `url_patterns` must accept either.
- Chapters are `/txt/<book_id>/<chapter_id>`, ids sequential (`下一章` → `…38369378`).
- Only `www.69shuba.com` verified. `.cx`/`69shu.com` also 403 but were never seen past
  the challenge — do not assume the same markup.

**Structure (8-14):**
- Metadata: the info page carries **full OpenGraph tags** — `og:novel:book_name`,
  `og:novel:author`, `og:description`, `og:image`, `og:novel:latest_chapter_name`.
  Use them (the `ixdzs` approach), not CSS. Note `og:description` embeds literal
  `<br />` needing cleanup.
- TOC: `#catalog ul li a[href]`.
- Content: `.txtnav` — **`<br>`-separated text nodes, zero `<p>`** → the `xbanxia`
  extraction shape, not `ixdzs`.
- Chrome inside `.txtnav` to decompose: `h1.hide720`, `div.txtinfo.hide720`,
  `div#txtright`, `div.contentadv`, `div.bottom-ad`.
- **TOC does NOT paginate.** 199 links, and `og:novel:latest_chapter_name` is
  `199.第199章 …` — counts match, so the whole TOC is on one page. No pagination loop
  (plan §2 Step 6's `medoctruyen` fallback is not needed).
- **Title-dup strip needs care.** The first body line duplicates the title, but
  *not exactly*: TOC/h1 gives `1.第1章 …` while the body line is `第1章 …`. The site's
  own titles are inconsistent — ch.1 is `1.第1章 …` but ch.2 is plain `第2章 …`.
  `xbanxia`'s `lines[0] == ref.title` would silently fail to strip. Needs a tolerant
  match (e.g. `ref.title.endswith(line)` / index-prefix normalisation).
- No obfuscation seen. Chapter 1 is ~2,102 chars against `40.75万字`/199 ≈ 2,048
  average — consistent with full content, no preview gate or paywall.

**Cloudflare (15-20):**
- (15) **No** — clearance does not survive outside the browser; none is issued.
- (16) Challenge = HTTP 403 + `cf-mitigated: challenge`, ~5.6KB UTF-8 interstitial.
- (17) 403 every time across 4 hosts; no 503/200 variant seen.
- (18) N/A — no cookie, so no lifetime. Re-solve concerns are moot; the browser session
  is the clearance.
- (19) **Passive** for real Chrome — cleared with no interaction. Headed still
  recommended, but no checkbox was required in this run.
- (20) Only `zh_choose`; no Cloudflare cookies at all.

## [STEP 3] Browser session — headless is fingerprinted — 2026-07-16
Plan 015.02 §3 Step 3. Added `src/noveltrans/cf_browser.py` + `tests/test_cf_browser.py`.

### Sub-step 3a VERDICT: headless is fingerprinted — headed default stands
Ran `scripts/capture_69shuba.py --headless` against the real site. It sat on the
challenge for the full **180s timeout and never cleared**, where headed clears
instantly. So Cloudflare fingerprints headless Chrome here, and the headed default is
now settled **on evidence rather than analogy** (015.02 §5 wanted exactly this).

Consequence: the visible-window UX imposition is unavoidable, not a preference. The
`headless` parameter stays exposed (a future CF policy change could make it viable)
but will very likely fail today; the module docstring records the measurement so the
next person doesn't re-run it.

### Sub-step 3b: `challenge.html` fixture — and a bug in the capture of it
First attempt saved a **14-byte 404** as the challenge fixture. Cause: the request
sent `User-Agent: python-requests`, and the site answers an obvious bot UA with a bare
404 rather than the Cloudflare interstitial. Sending `base.USER_AGENT` returns the real
thing: **403, `cf-mitigated: challenge`, 5,652 bytes**. Fixed, and `capture_challenge_
fixture()` now refuses to save anything that isn't an interstitial — a 14-byte fixture
would have made `looks_like_challenge` tests pass while proving nothing.

### `cf_browser.py`
- `BrowserSession` — one persistent context per scan/download batch, **not** a
  `solve()` returning a `Clearance`: there is no token, so the live session *is* the
  clearance (015.02 §2).
- **Lazy launch** — constructing a session never starts Chrome (adapters get built for
  cancelled scans and in tests).
- **One page, one `page.route()`** for the session's life; not re-paid per navigation.
- `_BLOCKED_TYPES = {image, font, media, stylesheet}` — never `document`/`script`/`xhr`,
  since CF's check is JS-driven. Kept a module constant so emptying it is the first
  diagnostic if 403s appear.
- `_throttle()` — copies `HttpClient._throttle`'s shape. **The only thing between a
  199-chapter batch and hammering the site**, since `HttpClient`'s throttle is bypassed
  entirely on this path.
- `looks_like_challenge(markup)` — pure, no Playwright import; folded in here rather
  than a `scrapers/cloudflare.py` module, per 015.02 §2 (one predicate, one caller).
- Navigation failure (closed window / crash) closes the session and raises, so a dead
  browser isn't retried 198 more times.
- Docstring records the measured facts and the do-not-parallelise warning, following
  `discord_unlock.py`'s candour precedent.

### Tests: 16 new, none launch a browser
`BrowserSession` takes `sync_playwright` injectably — Step 1's design paying off.
Covers: lazy launch, one-context-reuse across navigations, route registered once,
throttle between navigations, challenge→raise (against the **real** interstitial),
content→no false positive (against all three real content fixtures), navigation
failure closes the session, `close()` idempotent/pre-launch/context-manager, and
`.cf-profile` ≠ `.discord-profile`.

Self-review caught one junk test: `assert ".txtnav" in session.get_html(…) or True` —
vacuously true, and its second assertion tested the fixture rather than the code.
Rewritten to assert `get_html` returns the markup verbatim.
