# 015.01 — Initial Plan: 69shuba.com source support

## 0. Summary

69shuba sits behind a **Cloudflare managed challenge** (403 + `cf-mitigated: challenge`), so the plain `requests`-based adapter shape used by `xbanxia`/`ixdzs` cannot work. Per the user's decision, Playwright solves the challenge.

The recommended design is **hybrid**: drive Playwright **once** to solve the challenge on a persistent profile, harvest `cf_clearance` **plus the exact User-Agent that solved it**, inject both into the existing `HttpClient`, and do all bulk chapter fetching over fast plain `requests`. Re-solve via browser only when a response comes back challenged again.

**Because the real HTML has never been seen, every CSS selector, URL shape, encoding and pagination claim in this plan is a placeholder.** Step 2 (capture real HTML) must complete before Step 4 (write selectors) is even started. Section 6 lists what must be verified.

Work splits into: a shared browser module, a CF-detection module, a CF solver, small `base.py` additions (UA override + forced encoding), the adapter itself, a fixture-capture script, and tests.

---

## 1. Recommended architecture

```
scan/download (QThread worker)
  └─ Shuba69Adapter._get_html(url)
       ├─ HttpClient.get_html(url, encoding="gb18030")   ← fast path, ~all requests
       │    └─ base.py raises ChallengeError on cf-mitigated (no retry)
       └─ on ChallengeError:  cf_browser.solve(force=True)   ← slow path, rare
            ├─ Playwright → persistent profile → 69shuba.com
            ├─ harvest cookies + page.evaluate("navigator.userAgent")
            ├─ client.set_cookies(...) ; client.set_user_agent(...)
            └─ retry the fetch ONCE; a second ChallengeError propagates
```

### Why hybrid over always-Playwright

| | Hybrid | Always-Playwright |
|---|---|---|
| 1000-chapter novel | ~1 browser launch + 1000 `requests` GETs | 1000 page navigations; each loads JS/CSS/images |
| Throttle / retry / backoff | reuses `HttpClient` | must be reimplemented |
| Encoding handling | reuses `get_html` | Playwright decodes for us (different code path) |
| Parallel-download compatibility | works (it's just `requests`) | ~impossible (browser contexts are heavy) |
| RAM | ~0 during bulk | one Chromium alive for the whole batch |
| Fragility | needs UA+cookie harvesting to be exactly right | fewer moving parts |
| Failure mode | cookie expiry mid-batch → re-solve | browser crash mid-batch → batch dies |

**Recommendation: hybrid.** Always-Playwright is genuinely simpler, and it is a defensible choice if the hybrid's cookie-harvesting turns out to be flaky in practice. But a browser navigation is roughly 1–3s versus ~50ms for a `requests` GET; on a 1000-chapter novel that is the difference between ~40 minutes and ~25 minutes of *pure overhead* on top of the 1.5s politeness delay — and it discards the throttle/retry/encoding machinery that already works. The hybrid's extra complexity is concentrated in one module (`cf_browser.py`) and one `base.py` addition, both testable.

**Fallback if hybrid fails:** if `cf_clearance` proves unusable outside the browser (see §5 R1), the adapter's `_get_html` is the *only* thing that changes — swap its body for a browser navigation. The adapter's parsing, the tests, and the fixtures are all unaffected. This is why `_get_html` is a seam.

### Where the re-solve loop lives

**In the adapter's `_get_html()`, not in `HttpClient`.** This mirrors the existing `ixdzs._get_html()` precedent exactly (`ixdzs.py:38-46` transparently solves that site's JS challenge inside a `_get_html` wrapper).

Critically, it must **not** fight `HttpClient`'s retry loop. `base.py:53-65` retries only on `requests.RequestException`. `ChallengeError` is a `ScrapeError`, *not* a `RequestException`, so raising it from inside `_request` propagates **immediately** past the `for attempt in range(...)` loop with zero backoff sleeps. That is the correct behaviour: retrying a challenged request with the same UA and cookies is guaranteed to fail 3 more times and waste 3 seconds. The re-solve is a *different* action at a *higher* level, so it belongs a layer up.

### Step 3 — CF detection + error type

**`src/noveltrans/errors.py`** — add:
```python
class ChallengeError(ScrapeError):
    """The site returned a bot-challenge interstitial instead of content."""
```
It subclasses `ScrapeError`, so `DownloadWorker`'s `except NovelTransError` (`workers.py:799`) still catches it and marks the row — no worker changes needed for correctness.

**`src/noveltrans/scrapers/cloudflare.py`** — NEW, pure, no Playwright import, fully unit-testable:
- `is_challenge_response(response) -> bool` — primary signal `response.headers.get("cf-mitigated") == "challenge"`; secondary `status_code in (403, 503)` **and** an interstitial marker in the body (`challenges.cloudflare.com`, `cf-browser-verification`, `__cf_chl` — VERIFY which actually appear).
- `is_challenge_html(html) -> bool` — for the case where a 200 still carries an interstitial.

Keep the header check primary and the body check a guard: a plain 403 for an unrelated reason must **not** be mistaken for a challenge, or we'd launch a browser pointlessly on every real 403.

### Step 4 — `base.py` additions

Three changes to `src/noveltrans/scrapers/base.py`. All are additive with defaults preserving current behaviour for the four existing adapters.

**4a. UA override.** `HttpClient.__init__(..., user_agent: str = "")` defaulting to the module `USER_AGENT`, plus:
```python
def set_user_agent(self, user_agent: str) -> None:
    """Override the default UA. cf_clearance is bound to the UA that solved the
    challenge — a mismatch silently invalidates the cookie."""
```
This is non-negotiable for the hybrid: `cf_clearance` is bound to the UA **and** IP that solved it. The hardcoded Chrome 126 UA will not match whatever Chrome the user actually has, and the failure is *silent* (403, not an error message).

**4b. Challenge detection inside `_request`.** Currently `raise_for_status()` (line 60) turns the 403 into an `HTTPError`, which the loop retries 3× and then wraps in a generic `ScrapeError` — the `cf-mitigated` header is lost. Change to:
```python
response = self._session.request(...)
if is_challenge_response(response):
    raise ChallengeError("Cloudflare challenge returned instead of content", url)
response.raise_for_status()
```
`ChallengeError` isn't a `RequestException`, so it escapes the retry loop immediately (as designed, §1). Blast radius: other adapters would now get `ChallengeError` instead of `ScrapeError` if their site ever put up CF — strictly an improvement, and `ChallengeError` is a `ScrapeError` so nothing breaks.

> Alternative considered: a `challenge_check` callback passed per-client, keeping `base.py` CF-agnostic. Rejected as over-engineering — Cloudflare is generic web infrastructure, not a 69shuba quirk, and one hardcoded check is cheaper than a hook nobody else uses.

**4c. Forced encoding.** `get_html(url, *, encoding: str = "", **kwargs)`:
```python
if encoding:
    response.encoding = encoding
elif response.encoding in (None, "ISO-8859-1"):
    response.encoding = response.apparent_encoding
```
Note `encoding` must be **keyword-only** — `**kwargs` is forwarded to `requests`, so a positional would collide.

**Why the `apparent_encoding` fallback is probably not enough** (VERIFY): (a) if 69shuba declares `charset=utf-8` while serving GBK, the `in (None, "ISO-8859-1")` guard never fires and you get silent mojibake; (b) `apparent_encoding` is statistical and routinely reports `GB2312` for GBK content — GBK is a *superset*, so rare characters would then fail. Passing `encoding="gb18030"` (superset of both GB2312 and GBK, decodes all three) is the safe explicit choice. **But confirm the real charset from the Step 2 fixture first** — if it's genuinely UTF-8, drop this entirely.

### Step 5 — CF solver (`src/noveltrans/cf_browser.py`) — NEW

Sibling of `discord_unlock.py`, following its documented conventions.

```python
@dataclass(frozen=True)
class Clearance:
    cookie_string: str   # "cf_clearance=…; …" for HttpClient.set_cookies
    user_agent: str      # the UA that solved it — MUST travel with the cookie
```

- `profile_dir() -> Path` → `DEFAULT_LIBRARY_DIR / ".cf-profile"` — **a separate profile from `.discord-profile`.** Do not share: different sites, different cookie jars, and a Discord login has no business in a scraping profile.
- `solve(url, *, headless=False, timeout_ms=120_000) -> Clearance` — launch persistent context, `page.goto(url)`, wait for the challenge to clear, then harvest:
  - `context.cookies()` → filter to the 69shuba domain → join into a `name=value; …` string.
  - `page.evaluate("() => navigator.userAgent")` → the **real** UA. Do not guess it, and do not read it from the launch args.
  - Verify success before returning: check the page is real content, not the interstitial. Returning a bogus `Clearance` would poison every subsequent request with a silent 403 loop.
  - `finally: close(context, playwright)`.
- **Module-level `threading.Lock`** around `solve()`. `DownloadWorker.run()` is currently sequential (`workers.py:781`, a plain `for chapter in pending`), so no lock is needed *today* — but feature 014's parallel direction is live, and two threads each launching Chromium on the same profile dir would corrupt it. The lock is ~3 lines; add it now.
- A module-level cache of the last `Clearance` so a second adapter instance in the same process reuses it rather than relaunching.

**Headed vs headless: default to headed (`headless=False`).** `discord_unlock.py` already documents that headless "invites a mid-run re-challenge" because of the `HeadlessChrome` UA and distinct fingerprint. Cloudflare fingerprints *substantially* harder than Discord. Headed also lets the user click an interactive checkbox if CF escalates — headless would just hang until timeout. The window is brief and appears once per session. Expose `headless` as a parameter and a Settings toggle for users who want to try it.

**Threading:** Playwright's sync API cannot run in a thread with an active asyncio loop. It's safe here for the same reason `UnlockWorker` is (`workers.py:814-846`, which already does exactly this): QThreads have a Qt event loop, not an asyncio one. Never call `solve()` from the GUI thread — it blocks for seconds to minutes.

### Step 6 — The adapter (`src/noveltrans/scrapers/shuba69.py`) — NEW

⚠️ **Module name cannot be `69shuba.py`** — `69shuba` is not a valid Python identifier, so `from noveltrans.scrapers import 69shuba` in `_import_adapters()` is a `SyntaxError`. Verified. Use:
- module `shuba69.py`, class `Shuba69Adapter`
- `name = "69shuba"` — a plain string, unconstrained; this is what lands in `NovelMeta.site` and the fixture dir
- `display_name = "69书吧 (69shuba.com)"` (VERIFY the site's own Chinese name)

```python
url_patterns = [r"69shuba?\.(?:com|cx)/book/\d+"]   # VERIFY the domain set
ENCODING = "gb18030"                                 # VERIFY
SEL_TITLE = "…"       # PLACEHOLDER — fill from the Step 2 fixture
SEL_AUTHOR = "…"      # PLACEHOLDER
SEL_DESCRIPTION = "…" # PLACEHOLDER
SEL_TOC_LINKS = "…"   # PLACEHOLDER
SEL_CONTENT = "…"     # PLACEHOLDER
```
`matches()` uses `re.search`, so trailing slashes and `.htm` suffixes both match without extra work.

**`_get_html(url)`** — the one interesting method:
```python
def _get_html(self, url: str) -> str:
    try:
        return self.client.get_html(url, encoding=self.ENCODING)
    except ChallengeError:
        self._apply_clearance(force=True)   # solve + set_cookies + set_user_agent
        try:
            return self.client.get_html(url, encoding=self.ENCODING)
        except ChallengeError as exc:
            raise ChallengeError(
                "Không vượt được kiểm tra Cloudflare của 69shuba…", url
            ) from exc
```
Retry **exactly once**. A second challenge after a fresh solve means something systemic (IP flagged, cookie unusable outside the browser) that a third attempt won't fix.

`fetch_metadata` / `fetch_chapter_list` / `fetch_chapter` then follow the `xbanxia` template verbatim: `ScrapeError("… — page layout may have changed", url)` on missing selectors, strip the title-duplicate first line, `"\n\n".join(lines)`, `ScrapeError("Chapter content is empty", ref.url)` on empty.

`source_lang` stays `"zh"` (the `NovelMeta` default) — 69shuba is Chinese, unlike medoctruyen's `"vi"`.

**Pagination — genuinely unknown.** If the TOC paginates, follow the `medoctruyen` precedent (`medoctruyen.py:118-148`): find the last page number, loop, and break when a page yields zero chapter links, with a `_MAX_TOC_PAGES` sanity cap. If it doesn't, do nothing. VERIFY.

## 3. Testing strategy

**Tests must never launch a browser.** CI has no Chrome, no `playwright install`, and no unblocked route to 69shuba.

| File | What | How |
|---|---|---|
| `tests/test_cloudflare.py` | `is_challenge_response` / `is_challenge_html` | Pure. Synthetic responses: `cf-mitigated` 403 → True; plain 403 → **False**; 200 content → False. |
| `tests/test_scrapers_shuba69.py` | The adapter | `@responses.activate` + byte fixtures. `TestRegistry` / `TestMetadata` / `TestChapterList` / `TestChapterContent`, per `test_scrapers_xbanxia.py`. Plus a `TestChallenge` class asserting that a `cf-mitigated` 403 raises `ChallengeError` and that `_get_html` calls the solver — with the solver **monkeypatched**, never real. |
| `tests/test_scrapers_base.py` | The `base.py` additions | `set_user_agent`, keyword-only `encoding`, challenge-not-retried (assert the retry loop did **not** sleep 3×). |
| `@pytest.mark.live` | Real site | Excluded by default via `addopts = "-m 'not live'"`. Run manually with `make test-live`. |

**The fixture encoding problem — worth being explicit about.** `conftest.load_fixture` reads `encoding="utf-8"` and returns `str`. If tests feed a UTF-8 string body to `responses` while the adapter forces `gb18030`, the decode path is never exercised and the test is a lie.

Fix: save fixtures as **raw GBK bytes** (Step 2) and add a sibling to `conftest.py`:
```python
def load_fixture_bytes(site: str, name: str) -> bytes:
    return (FIXTURES_DIR / site / name).read_bytes()
```
Leave the existing `load_fixture` alone — four other test modules depend on it. Then `responses.get(URL, body=load_fixture_bytes("69shuba", "novel.html"))` genuinely round-trips the encoding, and the test would catch a GBK regression. This is the *one* fixture decision that's hard to reverse later, so get it right in Step 2.

## 4. Packaging (`packaging/NovelTrans.spec`) — OPEN QUESTION, needs a decision

There is an existing hole this feature makes worse. Playwright is an optional extra and `make app` installs only base + tts, so **in the frozen `.app`, `import playwright` fails**. `_require_playwright()` then tells the user to run `pip install 'noveltrans[discord]'` — but there is no pip in a bundled `.app`. That's a dead end today for Discord auto-unlock, and it would make 69shuba simply not work in the shipped app.

Options:
1. **Add `playwright` to the frozen build** — `collect_all("playwright")` in the spec + install it in the `make app` target. Relies on the `channel="chrome"` fallback so the user's existing Chrome is driven and no Chromium download is needed. Adds maybe ~10MB. **Does not work if the user has no Chrome.**
2. Bundle Chromium too — ~150MB+ into the `.app`, and `playwright install` writes to `~/Library/Caches/ms-playwright`, not into the bundle. Painful; feature 012's bundling work is the relevant precedent for how fiddly this gets.
3. Ship without it and show a clear Vietnamese error telling the user to install Chrome + run the app from source for 69shuba.

**Lean toward (1)**, but this needs the user's call — it changes what "the app works" means for anyone without Chrome. Do not let this block Steps 1–8; the source-run path works regardless.

## 5. Risks & open questions

**R1 — `cf_clearance` may not work outside the browser (highest risk).** Cloudflare increasingly binds clearance to a TLS/JA3 fingerprint, not just UA+IP. `requests` has a different TLS fingerprint from Chrome, so the harvested cookie may be rejected no matter how perfectly UA and cookies are copied. **Step 2 tests this before any adapter code exists** — if the byte-capture GET returns 403, the hybrid is dead and the fallback is always-Playwright inside `_get_html` (§1). Everything else in the plan survives that pivot.

**R2 — Cookie is IP-bound.** VPN toggles, network changes, or a DHCP lease change mid-batch invalidate it. Handled: the next fetch raises `ChallengeError` → re-solve.

**R3 — CF escalates to an interactive challenge.** Headed mode lets the user click through; headless just hangs. Reinforces the headed default.

**R4 — Rate limiting.** Keep the default 1.5s `delay_seconds` and **do not** parallelize 69shuba downloads. Hammering a CF-protected site is the fastest way to a permanent IP block, which would also break the Playwright path.

**R5 — The refactor in Step 1 touches working code (feature 004).** Mitigated by running `make test` immediately after Step 1, in isolation.

**R6 — Legal/ToS.** Cloudflare challenges are an access control the site operator chose to deploy; this feature routes around it. The repo already carries a candid precedent — `discord_unlock.py`'s docstring states plainly that automating a Discord user account violates ToS and advises keeping usage low-frequency. Same honesty belongs in `cf_browser.py`'s docstring: a note that 69shuba deliberately deployed this challenge, that the polite delay must stay, and that this is intended for personal reading volume. Not a blocker, and a judgement call that's the user's to make — but it should be written down rather than left implicit, exactly as feature 004 did.

**R7 — Site drift.** Selectors written against one captured snapshot. The `@pytest.mark.live` tests are the early-warning system; `make test-live` exists for this.
