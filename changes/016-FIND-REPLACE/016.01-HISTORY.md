## [DECISIONS] Scope settled before planning — 2026-07-17
Asked the user the three questions that materially change the design:

1. **Target text: translated AND original** (both preview panes). Flagged the hazard —
   editing original source text is not persisted today and can desync from a
   re-download — the user chose it anyway, so the plan must handle it.
2. **Safety: preview matches, then apply.** Show a match count + per-chapter breakdown
   ("47 matches in 23 chapters") and require confirmation before writing. No undo.
3. **Matching: plain text + case-sensitivity toggle.** No regex, no whole-word.

### Verified codebase facts that shape the plan
- `project.edit_translation(idx, title, text)` exists and preserves engine/lang/status
  — the right foundation for writing translated replacements.
- **No `edit_content` method exists** — the original pane's edits aren't persisted at
  all today. "Find/replace on original" therefore needs a new storage method, and it
  must NOT flip `status` the way `save_content` does (a replace shouldn't reset a
  translated chapter to "downloaded").
- `replace_toc` preserves content/translation on re-scan, so a re-scan won't clobber
  edited originals; only a re-*download* (`save_content` on `content = ''`) would.
- The tab already has an editable side-by-side preview (`original_view` /
  `translated_view`) with save-on-blur / save-on-row-change via `_save_preview_edits`
  — find/replace must cooperate with that flow, not fight it.

(`find_replace.py`), two new storage methods (`edit_content`, `apply_replacements`,

- **Default fields: translated body + translated title ON**; both original-side fields
  opt-in.
- **Per-chapter scope = the previewed chapter** (`_preview_idx`, driven by selection).

## [STEP 1] Pure find/replace core — 2026-07-17
Plan 016.01 Step 1. Added `src/noveltrans/find_replace.py` + `tests/test_find_replace.py`.
No Qt, no sqlite — imports only `re`, `dataclasses`, and `Chapter`.

- `apply_to_text(text, search, replacement, case_sensitive) -> (new, count)`,
  `scan(chapters, …, fields) -> list[ChapterMatch]`, plus `total_matches` /
  `chapter_count` helpers. `FieldChange`/`ChapterMatch` are frozen dataclasses; each
  `FieldChange` carries the computed `new`, so the dialog applies exactly what was
  previewed without re-searching.
- Field constants (`FIELD_TITLE` / `CONTENT` / `TRANSLATED` / `TRANSLATED_TITLE`) are
  BOTH `Chapter` attribute names AND chapters-table column names — they coincide for
  these four, so the same string reads here and writes in `apply_replacements`. Uses
  `Chapter.index` (the model's attr; the DB column is `idx`, mapped at the storage
  boundary).
- **Two subtle correctness points, both tested:**
  - Case-insensitive uses `re.IGNORECASE` (a `str.replace` can't do case-insensitive).
  - Replacement is inserted via a **lambda replacer** (`subn(lambda _m: replacement)`),
    not a bare string — verified live that a bare-string `re.sub` reads `\g<0>` as a
    group reference and outputs `x` instead of the literal `\g<0>`. A user replacing a
    name with anything containing a backslash would otherwise get silent corruption.
- Empty search and empty field list both short-circuit to no-op / `[]`.

Tests: 14, all pure. Basic/case/empty/loop-safety/backreference-literal for
`apply_to_text`; ordering, per-field summing, field-selection exclusion, blank-title
label, and the helpers for `scan`.

Verified: `343 passed, 5 deselected` (was 329); `ruff check` clean.

## [STEP 2] Storage methods — 2026-07-18
Plan 016.01 Step 2. Added to `src/noveltrans/storage/project.py`, tests in
`tests/test_storage.py`.

- **`EDITABLE_COLUMNS`** — module constant `frozenset{"title","content","translated",
  "translated_title"}`. Required because `apply_replacements` interpolates column names
  into SQL (values stay bound params). `title` is in it per the user's opt-in, despite
  reverting on a TOC re-scan (the GUI warns).
- **`edit_content(idx, text)`** — the missing counterpart to `edit_translation`. Sets
  `content` + `updated_at` only; deliberately does NOT touch `status`/`error` the way
  `save_content` does — flipping a translated chapter to DOWNLOADED would re-queue it in
  `pending_translation`. That contrast is the point, and is the key assertion in the test.
- **`apply_replacements({idx: {col: new}})`** — the batched all-chapters write. One
  `with self._db:` transaction for the whole batch, `updated_at` bumped, `status`/`error`
  never touched. Whitelist-checks every column and raises `ValueError` on anything else.

### Verified the atomicity guarantee, not just assumed it
The plan claimed "one transaction → no half-applied replace". Confirmed it holds: a batch
where ch.0 has a valid write and ch.1 has a bad column raises `ValueError` and leaves
ch.0 **unchanged** — sqlite's `with conn:` rolls back the already-executed valid UPDATE.
Locked in as `test_a_bad_column_rolls_back_valid_writes_in_the_same_batch`.

Tests: 8 new (edit_content keeps status/translation, bumps updated_at; apply_replacements
multi-column/multi-chapter, status untouched, original-title writable, non-editable column
rejected, empty-changes no-op, mid-batch rollback).

Verified: `351 passed, 5 deselected` (was 343); `ruff check` clean.

## [STEP 3] FindReplaceDialog — 2026-07-18
Plan 016.01 Step 3. Added `src/noveltrans/gui/find_replace_dialog.py` +
`tests/test_find_replace_dialog.py`. Pure logic stays in `find_replace.py`; this is Qt
wiring only.

- Search/replace `QLineEdit`s, "Phân biệt hoa/thường" case toggle, scope radios
  (Chương hiện tại / Tất cả chương — "current" disabled and "all" forced when
  `preview_idx is None`), four field checkboxes (translated body + title default on;
  original body + title opt-in), a `QListWidget` breakdown, and a summary label.
- **Preview gating**: Apply starts disabled, enables only after a preview finds > 0
  matches, and **any input change re-invalidates** (clears the cached scan, disables
  Apply). So what's applied always equals what was shown — the safety net the user
  chose in place of an undo.
- **Applies the cached scan verbatim** — `changes = {m.index: {c.field: c.new …}}` from
  the `ChapterMatch` list, so the write is exactly the previewed result, no re-search.
  `Chapter.index` maps from DB `idx` (verified), so it keys `apply_replacements` right.
- Original-title warning label toggles with its checkbox. Emits `applied(set)` with the
  changed indices so the tab (Step 4) can refresh only those rows.

## [STEP 4] Tab wiring — 2026-07-18
Plan 016.01 Step 4. `src/noveltrans/gui/tab_translate.py`; tests in

- **`_open_find_replace`** — guards (no project → info box; worker running → status
  message, mirroring `_retranslate_row`), then the load-bearing ordering:
  1. `_save_preview_edits()` flushes a half-typed manual edit to disk BEFORE the scan,
     so the replace sees latest text and a later focus-out can't clobber it;
  2. open the modal (which blocks further pane edits while up);
  3. on `applied`, `_on_replacements_applied` refreshes the table and reloads the open
     chapter — but only if its index is in the changed set. `_load_preview` resets the
     modified flag, so the reload doesn't trigger a redundant save-on-blur.
