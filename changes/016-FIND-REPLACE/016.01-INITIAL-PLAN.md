# Implementation Plan: Find-and-Replace for the Dịch (Translate) Tab

## Summary

Add a modal "Tìm & thay thế" dialog to the Dịch tab that does literal (non-regex) substring find-and-replace across chapter text, with:
- a **case-sensitivity** toggle,
- **field selection** (original body / translated body / translated title),
- **two scopes**: current previewed chapter vs. all chapters,
- a mandatory **preview** step ("47 khớp trong 23 chương" + per-chapter breakdown) before any write.

The work splits into three clean layers: a **pure, GUI-free core** (`find_replace.py`) that counts and applies replacements over `Chapter` objects; **two new storage methods** on `NovelProject` that persist edits without disturbing translation status; and a **QDialog** wired into `TranslateTab` that cooperates with the existing save-on-blur flow. No threads, no network, no browser — the string work over a 199-chapter novel is sub-millisecond in memory.

## Recommended UI + rationale

**A modal dialog** (`FindReplaceDialog`), launched from a new `QPushButton("Tìm & thay thế")` added to the existing `bottom_row` in `tab_translate.py`, next to "Dịch lại từ đầu".

Why a dialog rather than an inline toolbar row:
- The required per-chapter breakdown ("Chương 12: 3 khớp", …) needs a scrollable list; that does not fit inline under the existing buttons without crowding the splitter.
- Preview-then-confirm is inherently a modal, staged interaction (type → preview → confirm apply). A modal also naturally **blocks manual preview-pane editing while open**, which removes the race between a replace-all and the save-on-blur handler.
- Keeps the tab's primary surface (table + side-by-side panes) unchanged.

Dialog contents:
- `QLineEdit` "Tìm:" (search) and `QLineEdit` "Thay bằng:" (replacement).
- `QCheckBox` "Phân biệt hoa/thường" (case-sensitive), default **unchecked**.
- Scope as a `QButtonGroup` of two radios: "Chương hiện tại" (per-chapter) and "Tất cả chương" (all-chapters). The per-chapter radio is disabled (and all-chapters selected) when no chapter is previewed.
- Field `QCheckBox`es: "Bản gốc" (original `content`), "Bản dịch" (translated `translated`), "Tên chương dịch" (`translated_title`). Default: Bản dịch + Tên chương dịch checked.
- `QPushButton("Xem trước")` → runs the pure scan, fills a `QListWidget` breakdown and a summary `QLabel` ("47 khớp trong 23 chương" / "Không tìm thấy kết quả nào").
- `QPushButton("Thay thế")` (apply) + `QPushButton("Đóng")`. Apply is **disabled** until a preview has been run that found > 0 matches, and re-disabled whenever any input changes (so the user can't apply against a stale preview).

The whole button is disabled while a `TranslateWorker` is running (add it to `_reset_buttons()` / the enable-disable block in `_start_translate`).

## Field-scope recommendation (needs one user call)

Operate on **DB columns directly**, not on the concatenated pane text (the preview panes show `title\n\ncontent`; working on columns is cleaner and avoids re-parsing that join).

- **Recommended in scope:** `translated` (translated body), `translated_title`, and `content` (original body).
- **Recommended OUT of scope by default:** the original `title` column. Rationale: `replace_toc()` overwrites `title`/`url` on every re-scan (Tab 1), so an edit to the original title is silently reverted the next time the user re-scans the TOC — a footgun. `translated_title` and `content` are preserved by `replace_toc` (it only touches title/url), so they are safe. This is the one genuine decision to confirm — see Open Questions.

## The pure count/replace core (new file)

**`src/noveltrans/find_replace.py`** — no Qt, no sqlite, imports only `re`, `dataclasses`, and `noveltrans.models.Chapter`.

Field constants map to `Chapter`/DB column names:
```
FIELD_CONTENT          = "content"           # original body
FIELD_TRANSLATED       = "translated"        # translated body
FIELD_TRANSLATED_TITLE = "translated_title"
```

Signatures:
```python
@dataclass(frozen=True)
class FieldChange:
    field: str        # one of the FIELD_* column names
    old: str
    new: str
    count: int        # occurrences replaced in this field

@dataclass(frozen=True)
class ChapterMatch:
    idx: int
    label: str                     # display, e.g. "Chương 12 — <title>"
    count: int                     # total across selected fields
    changes: list[FieldChange]     # only fields with count > 0

def apply_to_text(text: str, search: str, replacement: str,
                  case_sensitive: bool) -> tuple[str, int]:
    """Literal replace. Returns (new_text, num_replacements)."""

def scan(chapters: Iterable[Chapter], search: str, replacement: str,
         fields: Sequence[str], *, case_sensitive: bool) -> list[ChapterMatch]:
    """Preview pass. Returns only chapters with count > 0, in idx order."""

def total_matches(matches: list[ChapterMatch]) -> int
def chapter_count(matches: list[ChapterMatch]) -> int
```

Implementation notes that matter:
- Use `re` for **both** case modes so case-insensitive works: compile `re.compile(re.escape(search), 0 if case_sensitive else re.IGNORECASE)`.
- Do the replacement with a **lambda** replacer — `pattern.subn(lambda m: replacement, text)` — so backslashes / `\g<...>` in the replacement string are inserted literally rather than interpreted as group references. This is the one subtle correctness point.
- Empty `search` → `apply_to_text` returns `(text, 0)`; `scan` returns `[]`.
- `str.replace`-style single pass means a replacement containing the search string ("cat"→"cats") cannot loop — `subn` scans once. (Mirrors the existing literal-replace approach in `translators/names.py::apply_glossary`, but this feature stays independent of the glossary — see below.)

## Storage changes (`src/noveltrans/storage/project.py`)

Two new methods, both bumping `updated_at`, **never** touching `status` or `error`.

**1. `edit_content(idx, text)`** — the missing counterpart to `edit_translation`:
```python
def edit_content(self, idx: int, text: str) -> None:
    """Edit the original text WITHOUT touching status/error (unlike save_content)."""
    with self._db:
        self._db.execute(
            "UPDATE chapters SET content = ?, updated_at = ? WHERE idx = ?",
            (text, _now(), idx),
        )
```
Contrast with `save_content`, which sets `status=STATUS_DOWNLOADED` and clears `error`. A find/replace must NOT do that: flipping a translated chapter back to "downloaded" would wrongly re-queue it for translation in `pending_translation`.

**2. `apply_replacements(changes)`** — batched all-chapters apply in a single transaction:
```python
_EDITABLE_COLUMNS = {"content", "translated", "translated_title"}

def apply_replacements(self, changes: dict[int, dict[str, str]]) -> None:
    """changes: {idx: {column: new_value}}. One transaction; bumps updated_at,
    never changes status/error. Column names are whitelisted."""
    with self._db:
        now = _now()
        for idx, cols in changes.items():
            bad = set(cols) - self._EDITABLE_COLUMNS
            if bad:
                raise ValueError(f"non-editable column(s): {bad}")
            assignments = ", ".join(f"{c} = ?" for c in cols)
            self._db.execute(
                f"UPDATE chapters SET {assignments}, updated_at = ? WHERE idx = ?",
                (*cols.values(), now, idx),
            )
```
The whitelist is required because column names are string-interpolated into SQL; values stay parameterized. All 199 UPDATEs run inside one `with self._db:` transaction (fast, atomic — no half-applied replace).

`edit_translation` needs **no change** (it already preserves engine/lang/status). For per-chapter scope the dialog can call `apply_replacements` uniformly for one row too, so a single code path covers both scopes.

## Ordered implementation steps

1. **`src/noveltrans/find_replace.py`** (new): the pure core above. Zero Qt/sqlite deps.
2. **`src/noveltrans/storage/project.py`**: add `edit_content` and `apply_replacements` (+ the `_EDITABLE_COLUMNS` constant) in the "writes" section.
3. **`src/noveltrans/gui/find_replace_dialog.py`** (new): `FindReplaceDialog(QDialog)` taking `(project: NovelProject, preview_idx: int | None, parent)`. It:
   - builds inputs described above;
   - on "Xem trước": reads `project.chapters()` once, calls `find_replace.scan(...)`, caches the resulting `list[ChapterMatch]`, populates the breakdown list + summary label, enables/disables Apply;
   - on "Thay thế": builds `changes = {m.idx: {c.field: c.new for c in m.changes}}` from the **cached** scan result (so what was previewed is exactly what's applied), calls `project.apply_replacements(changes)`, then either accepts or emits a signal carrying the set of changed indices.
   - Scope handling: all-chapters passes every `Chapter`; per-chapter filters to `[project.chapter(preview_idx)]`.
4. **`src/noveltrans/gui/tab_translate.py`**:
   - add the `QPushButton("Tìm & thay thế")` to `bottom_row`, connect to `_open_find_replace`;
   - `_open_find_replace()` ordering (critical — see cooperation below);
   - include the new button in the enable/disable sets in `_start_translate` and `_reset_buttons`.
5. **Tests** (see strategy below).

## Cooperation with `_save_preview_edits` (exact ordering)

`_open_find_replace()`:
1. Guard: if `self.project is None`, show the same "Chưa chọn truyện" info box used elsewhere and return.
2. Guard: if a translate worker is running, bail with a status message (mirror `_retranslate_row`).
3. **`self._save_preview_edits()`** — flush any pending manual edit typed into `translated_view` to disk FIRST, so the scan/apply sees the latest text and a later focus-out can't clobber the replace.
4. Open the dialog modally with `self._preview_idx`. (The modal blocks further pane edits while open.)
5. After the dialog applies (via its signal or a truthy exec result), on the tab:
   - `self.model.set_chapters(self.project.chapters())` (or update only changed rows) to refresh table titles/status column;
   - if `self._preview_idx` is among the changed indices (or unconditionally, cheaply), reload the panes: `self._load_preview(self.project.chapter(self._preview_idx))`. `_load_preview` already calls `self.translated_view.document().setModified(False)`, so the reloaded pane won't trigger a redundant save-on-blur.

Because step 3 flushes and `_load_preview` resets the modified flag, a replace and a manual edit cannot clobber each other.

## Performance

Pure in-memory string ops. A 199-chapter novel with, say, a few KB per chapter is well under a millisecond for both the scan and the apply pass. **No worker thread needed** — say so explicitly. The dialog reads `project.chapters()` once for preview and reuses the cached `ChapterMatch` list for apply (the modal prevents intervening edits), so there is a single DB read and a single batched write transaction. If a pathologically large novel ever janked the UI, the scan could move to a `QThread`, but that is not warranted here; note it and move on.

## status / updated_at semantics

- **Bump `updated_at`**: the text genuinely changed; both new methods set it via `_now()`, consistent with every other write.
- **Never change `status` or `error`**: a find/replace is a text correction, not a re-download or re-translation. Flipping a translated chapter to "downloaded" would re-queue it in `pending_translation`; clearing `error` would hide a real failure. This is precisely why `edit_content` must differ from `save_content`, and why we do not reuse `save_translation`.

## Relationship to `translators/names.py` (glossary)

**Keep them independent.** The glossary (`build_glossary`/`apply_glossary`) is a *pre-translation, source-side* transform: it detects Chinese names and substitutes Hán-Việt readings in the original text before it reaches the Google engine, driven by `data/hanviet.json`. Find/replace is a *post-hoc, user-driven* correction over already-stored original/translated text, with arbitrary user-chosen strings and a preview/confirm gate. Different trigger, different data, different lifecycle. The only overlap is the literal `str.replace` mechanic — and `apply_to_text` intentionally mirrors that literal approach for consistency — but forcing integration (e.g. routing find/replace through the glossary) would couple an interactive editing feature to a CJK-specific pipeline for no benefit. Independent is cleaner; if the user later wants "add this correction to the glossary," that is a separate follow-up.

## Testing strategy

**Pure core — `tests/test_find_replace.py`** (no `qapp`, fast):
- `apply_to_text`: basic replace + count; case-sensitive vs. case-insensitive; empty search → `(text, 0)`; replacement containing search ("cat"→"cats") replaces once per match, no loop; replacement containing `\1`/`\g<0>` inserted literally (the lambda-replacer guard).
- `scan`: multi-chapter breakdown counts; only chapters with matches returned; multiple fields summed per chapter; field selection excludes unselected columns; `total_matches`/`chapter_count` helpers.
- Build `Chapter` objects directly (as `test_names.py`/model tests do) — no project needed.

**Storage — extend `tests/test_storage.py`** (follow the `edit_translation` precedents):
- `test_edit_content_keeps_status`: after `save_content`+`save_translation`, `edit_content` changes `content` but leaves `status == STATUS_TRANSLATED`, `error`, and all translation metadata intact (the key contrast with `save_content`).
- `test_edit_content_bumps_updated_at`.
- `apply_replacements`: multi-chapter multi-column write in one call updates exactly the given fields; rejects a non-whitelisted column with `ValueError`; leaves `status` untouched; empty `changes` dict is a no-op.

**Light GUI wiring — `tests/test_find_replace_dialog.py`** (uses the offscreen `qapp` fixture, like `test_widgets.py`/`test_main_window.py`):
- Construct a real `NovelProject` (via `library_dir`/`sample_meta`/`sample_refs` fixtures), seed content + a translation, instantiate `FindReplaceDialog`, drive it programmatically (set search/replace text, click preview, assert summary label and Apply-enabled state, click apply), then assert the DB reflects the replacement and `status` is unchanged. Keep the heavy logic in the pure core so this stays a thin wiring check.

## Risks / edge cases

- **Empty search string** → preview and apply disabled; no-op in the core regardless.
- **search == replacement**: if identical AND case-sensitive, apply is a pure no-op that still bumps `updated_at`; recommend disabling Apply in that exact case. With case-insensitive it may still change case (e.g. "bob"→"Bob"), so only disable on byte-identical + case-sensitive.
- **Zero matches** → Apply disabled, summary shows "Không tìm thấy kết quả nào."
- **Short/common search over 199 chapters** — the whole point of the preview: the count + per-chapter breakdown is the guard the user chose. No undo is provided (not requested), so the confirm step is load-bearing; make the Apply button clearly labeled and gated behind a fresh preview.
- **Stale preview**: invalidate the cached scan / disable Apply whenever any input (search, replace, case, scope, fields) changes, so the applied change always equals the previewed one.
- **Original-title clobber**: excluded by default because `replace_toc` reverts it on re-scan (documented rationale above).
- **Match spans a title/body boundary**: impossible — we replace within individual columns, never across the `title\n\ncontent` join shown in the panes.
- **Case-insensitive replace does not preserve matched case** — plain literal replacement inserts the replacement verbatim; call this out in the dialog's mental model (it is the agreed "plain text" behavior, not smart-case).

## Open questions — RESOLVED by the user (2026-07-17)

1. **Original chapter `title` in scope? → INCLUDE, with a warning.** The user chose to
   allow replacing the original source title too. Consequences for the build:
   - Add `title` to `_EDITABLE_COLUMNS` and add a fourth field checkbox "Tên chương gốc".
   - The dialog must show a visible note that a TOC re-scan (Tab 1) reverts original-title
     replacements, since `replace_toc()` overwrites `title`/`url` from the site. Put it as
     a muted `QLabel` near the field checkboxes, shown (or emphasised) when "Tên chương
     gốc" is ticked. This is the one field with surprising revert behaviour; the other
     three (`content`, `translated`, `translated_title`) are preserved across re-scans.
2. **Default field checkboxes → "Bản dịch" + "Tên chương dịch" ON**, "Bản gốc" and "Tên
   chương gốc" OFF. Matches the motivating use case (fix a name in the *translated*
   output). The two original-side fields are opt-in.
3. **Per-chapter scope target → the previewed chapter (`_preview_idx`).** `_preview_idx`
   is driven by table selection, so "current chapter" means the previewed one. Multi-row
   scope is out of stated requirements.
