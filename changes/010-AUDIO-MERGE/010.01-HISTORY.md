# 010 — Audio merge (whole-novel single file) — History

## [PLAN] 010.02-INITIAL-PLAN — 2026-07-15

Generated plan: `010.02-INITIAL-PLAN.md`

Merge all rendered per-chapter audio into one file from the Audio tab — default an
**M4B audiobook with per-chapter markers** (verified ffmpeg has the `aac` encoder),
with a plain **joined MP3** alternative. New `tts/merge.py` builds an ffmpeg concat
list + ffmetadata chapter file (offsets from stored `audio_seconds`); a `MergeWorker`
runs it off-thread; the tab gets a format combo + "Ghép thành 1 file" button.

Decisions:
- **Merge only chapters matching the selected voice** (in index order); warn if some
  chapters lack audio in that voice, then merge the rest.
- **Marker titles follow `audio_source`** (translated audio → translated title, original
  → original title).
- **No schema change** — reads existing audio fields. Offsets from stored `audio_seconds`
  (ffprobe-exact durations noted as the precise alternative).

### Updated (same day) — chapter selection modes

Per user: the merge chooses which chapters to combine — **Toàn bộ** / **Từ chương X đến
Y** / **Theo lô N chương** (batch → many files). Added a pure `plan_merge_windows(...)`
helper (the testable heart: filters by voice, splits into number-windows), made
`MergeWorker` emit multiple files, and added mode/range/batch controls to the tab.
Ranges/batches use chapter *numbers* (not audio-count) so boundaries are predictable and
gaps don't shift them. Naming: `<slug>.m4b` (all) or `<slug>-0001-0010.m4b` (range/batch).
This also resolves the "one huge file" worry (700 chapters → batch of 50 = 14 files).

## 2026-07-15 — Fix "batch merge stuck" + cancellation/progress

User reported batch merge "kind of stuck". **Root cause:** `MergeWorker.__init__` did
`self.start = start`, which shadowed `QThread.start()` — so `worker.start()` called the
range int/None instead of launching the thread. The merge never ran; the button stayed
disabled → looked stuck. (Unit tests missed it because they call `run()` directly, never
`start()`.) Fix: range params renamed to `start_num`/`end_num`/`batch_size`; added a
regression test asserting `MergeWorker.start` stays callable.

While here, addressed the poor UX that made a slow merge indistinguishable from a hang:
- `merge_chapters` now takes `cancelled` and runs ffmpeg via `Popen` (stderr→file to
  avoid a pipe-buffer deadlock), polling for cancel → `MergeCancelled`; early-cancel check.
- `MergeWorker` passes cancellation and treats `MergeCancelled` as a clean stop (keeps
  finished files).
- Tab: the "Dừng" button is enabled during merge and cancels it; the progress bar tracks
  windows done; finish/cancel reset the controls.

**Verification:** drove the *full* AudioTab batch merge through a Qt event loop
(auto-answering dialogs) → 3 M4B files produced, status "Đã ghép xong 3 file". 232 tests
pass, ruff clean.
