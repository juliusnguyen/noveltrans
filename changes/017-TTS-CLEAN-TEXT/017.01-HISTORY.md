## [DECISIONS] Settled before planning — 2026-07-18
1. **Whitelist strategy** — keep known-good (Vietnamese letters + diacritics, ASCII
   letters, digits, whitespace, prosody punctuation), strip everything else. A
   never-seen glyph is removed automatically.
2. **Settings toggle, default ON** — a config bool + a checkbox, following the
   `tts_workers` / `tts_use_translation` / keep-awake precedent.

### Verified pipeline facts that shape the plan
- `TtsEngine.synthesize_chapter` (`tts/base.py:81-82`) builds `f"{title}\n\n{body}"`
  then calls `split_sentences` — the **single chokepoint** every caller routes through
  (`workers.py:425` sequential + `:487` parallel pool, `tts/__main__.py:49` CLI, tests).
- **The keep-set must not break chunking:** `split_sentences` splits on `\n\n` and
  `_SENTENCE_RE` keys off `. ! ? …` and quote/paren chars. Cleaning must preserve those
  and `\n`, or it breaks sentence splitting downstream.
- **The Vietnamese trap:** tone marks are precomposed non-ASCII letters (ộ U+1ED9,
  ữ U+1EEF). A `[a-zA-Z]` / strip-non-ASCII whitelist would destroy the text. Needs a
  Unicode-category + Latin-script-aware predicate that keeps Vietnamese but drops CJK.
- Non-destructive: clean only the copy fed to the engine; `translated` in the DB stays.
- Config precedent: `AppConfig` bool props (`self._s.value(..., True, type=bool)`);
  Settings checkboxes wired in the dialog save block.

## [STEP 1] Pure cleaner — `clean_for_tts` — 2026-07-18
Added `src/noveltrans/tts/clean.py` + `tests/test_tts_clean.py`. Pure `str -> str`,
no engine/model. (Built ahead of saving the plan file — design was already settled by
the pre-plan decisions; reconcile with the plan doc when the agent's output lands.)

### The keep-predicate, verified against real characters before committing to it
Keep = Latin-script letters (`unicodedata.category` L* AND `name` starts "LATIN"),
combining marks (M*, for decomposed tone marks), decimal digits (Nd), whitespace, and
an explicit prosody-punctuation set. Verified live: `a ằ ộ ữ đ Đ 1` all KEEP,
`中 文 ★` all drop. So Vietnamese precomposed tone marks and đ/Đ survive while stray
CJK and decorative symbols go — the exact split the feature wants.

### Details that matter
- **Prosody punctuation kept** (`. ! ? … , ; : " ' “ ” ‘ ’ « » ( )`) is a superset of
  what `tts/base.py::_SENTENCE_RE` keys on, so cleaning can't break sentence chunking.
  A test runs the cleaner output back through `split_sentences` to prove it.
- **`\n\n` preserved** (paragraph structure `split_sentences` needs); runs of spaces
  collapsed, blank runs capped at one blank line.
- **Dropped visible symbol → space** (so "A★B" → "A B", not "AB"); **control/format/
  zero-width → removed entirely** (so a ZWSP inside a word vanishes without splitting
  it — tested with a real U+200B).
- **Dashes** (— – ― − -) normalised to a plain hyphen so dialogue dashes survive.

### Tests: 17, pure
The load-bearing one: a real Vietnamese paragraph (tone marks, đ, digits, dialogue,
`1234`) returns **byte-identical**. Plus idempotence, emoji/decorative/CJK/markdown/
zero-width removal, dash normalisation, whitespace tidy, symbol-only → empty, and the
downstream-chunking-survives check.

Verified: `385 passed, 5 deselected` (was 368); `ruff check` clean.

### Reconciliation with the plan (which landed after the cleaner was built)
The plan (`017.01-INITIAL-PLAN.md`) matches this implementation — same module, same
`_keep`/`_is_latin` predicate, same chokepoint decision (Step 2, `clean: bool=True`
inside `synthesize_chapter`). Two deltas:
- **NFC normalization** — the plan recommends a `unicodedata.normalize("NFC", text)`
  Step 0 (composes stray combining marks), flagged explicitly as *the user's call*.
  This build does NOT include it: the `M*`-keep branch already preserves decomposed
  tone marks, and real translator output is NFC so it's a practical no-op. Surfaced to
  the user; trivial to add if wanted.
- **Full-width digits** — the plan drops non-ASCII digits (`Nd and isascii()`); this
  build keeps all `Nd`. Cosmetic — full-width digits are ~absent from VN novel text
  and "a digit is a digit" is defensible either way. Left as-is.

## [STEP 5] Fullwidth punctuation fix + in-app preview — 2026-07-18
Prompted by the user asking "can I view the text after cleaning?". Demonstrating the
cleaner on a sample surfaced a real gap, so this step does both.

### Fix: fullwidth CJK punctuation was being STRIPPED, losing prosody
The demo showed `nhanh lên！` → `nhanh lên` — the fullwidth `！` (U+FF01) vanished
because the keep-set only had ASCII `!`. Since the novels are translated *from Chinese*,
fullwidth punctuation (`！？，。：；「」（）` etc.) leaks through constantly, and stripping
a sentence-ender throws away the exact pause the cleaner exists to preserve.
- Added `_PUNCT_MAP` in `clean.py` normalising fullwidth/CJK punctuation to its ASCII
  equivalent (`！`→`!`, `。`→`.`, `，`/`、`→`,`, `：`→`:`, `；`→`;`, `（）`→`()`,
  `「」『』《》`→`"`) **before** the keep-filter, so the pause survives as an in-keep-set
  ASCII mark. 5 tests, incl. one proving a normalised `！` is a real `split_sentences`
  boundary.
- This is a genuine behaviour fix, not cosmetic: without it, CJK-sourced audio would run
  on at every leftover fullwidth ender.

### Feature: "Xem trước văn bản" button in the Audio tab
Answers the original question — there was no way to see the cleaned text (it's a
transient in-memory copy). Added a button that shows the selected chapter's text
**exactly as the engine will receive it**: `_engine_text_for(chapter)` mirrors
`AudioWorker._title_text_for` + `synthesize_chapter` (same translated-vs-original
source, same title+body join, same clean toggle), shown in a read-only dialog with a
note saying whether cleaning was applied.
- Logic extracted to `_engine_text_for` so it's testable without the modal; 3 tests
  cover translated source + clean, original source, and clean-off (raw passthrough).
- Rendered the dialog to confirm: read-only, correct note, and text
  `"Chương 1\n\nNội dung rồi!"` for a `★【】😀 中 ！`-laden chapter.

Verified: `400 passed, 5 deselected` (was 393); `ruff check` clean.

## [STEP 6] User-editable "extra characters to remove" — 2026-07-18
User asked to "add the list of special characters in Settings and let me add more".

### The conceptual mismatch, surfaced before building
The cleaner is a **whitelist** (keep-known-good), so there is no finite strip-list to
show or extend — everything not kept is already dropped. A literal "add characters to
strip" box would be a near-total no-op (whatever you type is already stripped). Raised
this with the user and offered the genuinely useful control instead; they chose **"extra
characters to always REMOVE"**, whose only real effect is on characters the whitelist
normally KEEPS (punctuation like `()` or quotes the user doesn't want voiced).

### Implementation
- **`clean_for_tts(text, extra_remove="")`** — applied as a **final pass on the cleaned
  output**, so the user lists characters as they appear in the preview (a fullwidth `！`
  has already become `!` by then, so listing `!` works). Removed chars → space (like any
  dropped symbol, so words don't merge); whitespace in the list is ignored so a stray
  space can't nuke every space.
- Threaded through: `config.tts_clean_extra_remove` (str, default "") → `AudioWorker.
  clean_extra_remove` → `synthesize_chapter(clean_extra_remove=...)` → `clean_for_tts`;
  and into the preview via `_engine_text_for`.
- **Settings**: a `QLineEdit` "Bỏ thêm ký tự" under the clean checkbox, disabled (and
  re-enabled live) with it, tooltip pointing at the preview button to find what to paste.

### Tests: 11 new (409 total)
Cleaner: strips listed kept-punctuation, no-op for already-stripped, empty=default,
whitespace ignored, matches the cleaned (normalised) form. Settings: field round-trips,
disabled when cleaning off. Preview: extra_remove reaches `_engine_text_for`. Plus an
end-to-end check: a real `AudioWorker` with `clean_extra_remove="()"` hands the engine
`"Chương 1 Nói một rồi!"` — parens gone, `！`→`!`.

One test over-specified spacing (`a“b”c` → expected `abc`, got `a b c`): the
removed-char→space rule is correct and consistent with the base cleaner, so the test was
fixed, not the code.

Verified: `409 passed, 5 deselected` (was 400); `ruff check src tests` clean.
