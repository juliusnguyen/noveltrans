## [DECISIONS] Settled before planning ‚Äî 2026-07-18
1. **Whitelist strategy** ‚Äî keep known-good (Vietnamese letters + diacritics, ASCII
   letters, digits, whitespace, prosody punctuation), strip everything else. A
   never-seen glyph is removed automatically.
2. **Settings toggle, default ON** ‚Äî a config bool + a checkbox, following the
   `tts_workers` / `tts_use_translation` / keep-awake precedent.

### Verified pipeline facts that shape the plan
- `TtsEngine.synthesize_chapter` (`tts/base.py:81-82`) builds `f"{title}\n\n{body}"`
  then calls `split_sentences` ‚Äî the **single chokepoint** every caller routes through
  (`workers.py:425` sequential + `:487` parallel pool, `tts/__main__.py:49` CLI, tests).
- **The keep-set must not break chunking:** `split_sentences` splits on `\n\n` and
  `_SENTENCE_RE` keys off `. ! ? ‚Ä¶` and quote/paren chars. Cleaning must preserve those
  and `\n`, or it breaks sentence splitting downstream.
- **The Vietnamese trap:** tone marks are precomposed non-ASCII letters (·ªô U+1ED9,
  ·ªØ U+1EEF). A `[a-zA-Z]` / strip-non-ASCII whitelist would destroy the text. Needs a
  Unicode-category + Latin-script-aware predicate that keeps Vietnamese but drops CJK.
- Non-destructive: clean only the copy fed to the engine; `translated` in the DB stays.
- Config precedent: `AppConfig` bool props (`self._s.value(..., True, type=bool)`);
  Settings checkboxes wired in the dialog save block.

## [STEP 1] Pure cleaner ‚Äî `clean_for_tts` ‚Äî 2026-07-18
Added `src/noveltrans/tts/clean.py` + `tests/test_tts_clean.py`. Pure `str -> str`,
no engine/model. (Built ahead of saving the plan file ‚Äî design was already settled by
the pre-plan decisions; reconcile with the plan doc when the agent's output lands.)

### The keep-predicate, verified against real characters before committing to it
Keep = Latin-script letters (`unicodedata.category` L* AND `name` starts "LATIN"),
combining marks (M*, for decomposed tone marks), decimal digits (Nd), whitespace, and
an explicit prosody-punctuation set. Verified live: `a ·∫± ·ªô ·ªØ ƒë ƒê 1` all KEEP,
`‰∏≠ Êñá ‚òÖ` all drop. So Vietnamese precomposed tone marks and ƒë/ƒê survive while stray
CJK and decorative symbols go ‚Äî the exact split the feature wants.

### Details that matter
- **Prosody punctuation kept** (`. ! ? ‚Ä¶ , ; : " ' ‚Äú ‚Äù ‚Äò ‚Äô ¬´ ¬ª ( )`) is a superset of
  what `tts/base.py::_SENTENCE_RE` keys on, so cleaning can't break sentence chunking.
  A test runs the cleaner output back through `split_sentences` to prove it.
- **`\n\n` preserved** (paragraph structure `split_sentences` needs); runs of spaces
  collapsed, blank runs capped at one blank line.
- **Dropped visible symbol ‚Üí space** (so "A‚òÖB" ‚Üí "A B", not "AB"); **control/format/
  zero-width ‚Üí removed entirely** (so a ZWSP inside a word vanishes without splitting
  it ‚Äî tested with a real U+200B).
- **Dashes** (‚Äî ‚Äì ‚Äï ‚àí -) normalised to a plain hyphen so dialogue dashes survive.

### Tests: 17, pure
The load-bearing one: a real Vietnamese paragraph (tone marks, ƒë, digits, dialogue,
`1234`) returns **byte-identical**. Plus idempotence, emoji/decorative/CJK/markdown/
zero-width removal, dash normalisation, whitespace tidy, symbol-only ‚Üí empty, and the
downstream-chunking-survives check.

Verified: `385 passed, 5 deselected` (was 368); `ruff check` clean.

### Reconciliation with the plan (which landed after the cleaner was built)
The plan (`017.01-INITIAL-PLAN.md`) matches this implementation ‚Äî same module, same
`_keep`/`_is_latin` predicate, same chokepoint decision (Step 2, `clean: bool=True`
inside `synthesize_chapter`). Two deltas:
- **NFC normalization** ‚Äî the plan recommends a `unicodedata.normalize("NFC", text)`
  Step 0 (composes stray combining marks), flagged explicitly as *the user's call*.
  This build does NOT include it: the `M*`-keep branch already preserves decomposed
  tone marks, and real translator output is NFC so it's a practical no-op. Surfaced to
  the user; trivial to add if wanted.
- **Full-width digits** ‚Äî the plan drops non-ASCII digits (`Nd and isascii()`); this
  build keeps all `Nd`. Cosmetic ‚Äî full-width digits are ~absent from VN novel text
  and "a digit is a digit" is defensible either way. Left as-is.

## [STEP 5] Fullwidth punctuation fix + in-app preview ‚Äî 2026-07-18
Prompted by the user asking "can I view the text after cleaning?". Demonstrating the
cleaner on a sample surfaced a real gap, so this step does both.

### Fix: fullwidth CJK punctuation was being STRIPPED, losing prosody
The demo showed `nhanh l√™nÔºÅ` ‚Üí `nhanh l√™n` ‚Äî the fullwidth `ÔºÅ` (U+FF01) vanished
because the keep-set only had ASCII `!`. Since the novels are translated *from Chinese*,
fullwidth punctuation (`ÔºÅÔºüÔºå„ÄÇÔºöÔºõ„Äå„ÄçÔºàÔºâ` etc.) leaks through constantly, and stripping
a sentence-ender throws away the exact pause the cleaner exists to preserve.
- Added `_PUNCT_MAP` in `clean.py` normalising fullwidth/CJK punctuation to its ASCII
  equivalent (`ÔºÅ`‚Üí`!`, `„ÄÇ`‚Üí`.`, `Ôºå`/`„ÄÅ`‚Üí`,`, `Ôºö`‚Üí`:`, `Ôºõ`‚Üí`;`, `ÔºàÔºâ`‚Üí`()`,
  `„Äå„Äç„Äé„Äè„Ää„Äã`‚Üí`"`) **before** the keep-filter, so the pause survives as an in-keep-set
  ASCII mark. 5 tests, incl. one proving a normalised `ÔºÅ` is a real `split_sentences`
  boundary.
- This is a genuine behaviour fix, not cosmetic: without it, CJK-sourced audio would run
  on at every leftover fullwidth ender.

Answers the original question ‚Äî there was no way to see the cleaned text (it's a

- Logic extracted to `_engine_text_for` so it's testable without the modal; 3 tests
