## [DECISIONS] Settled before planning — 2026-07-18
1. **Whitelist strategy** — keep known-good (Vietnamese letters + diacritics, ASCII
   letters, digits, whitespace, prosody punctuation), strip everything else. A
   never-seen glyph is removed automatically.
2. **Settings toggle, default ON** — a config bool + a checkbox, following the
   `tts_workers` / `tts_use_translation` / keep-awake precedent.

### Verified pipeline facts that shape the plan
- `TtsEngine.synthesize_chapter` (`tts/base.py:81-82`) builds `f"{title}\n\n{body}"`
  then calls `split_sentences` — the **single chokepoint** every caller routes through
  (`workers.py:425` sequential + `:487` parallel pool, `tts/__main__.py:49` CLI, tests).
- **The keep-set must not break chunking:** `split_sentences` splits on `\n\n` and
  `_SENTENCE_RE` keys off `. ! ? …` and quote/paren chars. Cleaning must preserve those
  and `\n`, or it breaks sentence splitting downstream.
- **The Vietnamese trap:** tone marks are precomposed non-ASCII letters (ộ U+1ED9,
  ữ U+1EEF). A `[a-zA-Z]` / strip-non-ASCII whitelist would destroy the text. Needs a
  Unicode-category + Latin-script-aware predicate that keeps Vietnamese but drops CJK.
- Non-destructive: clean only the copy fed to the engine; `translated` in the DB stays.
- Config precedent: `AppConfig` bool props (`self._s.value(..., True, type=bool)`);
  Settings checkboxes wired in the dialog save block.

## [STEP 1] Pure cleaner — `clean_for_tts` — 2026-07-18
Added `src/noveltrans/tts/clean.py` + `tests/test_tts_clean.py`. Pure `str -> str`,
no engine/model. (Built ahead of saving the plan file — design was already settled by
the pre-plan decisions; reconcile with the plan doc when the agent's output lands.)

### The keep-predicate, verified against real characters before committing to it
Keep = Latin-script letters (`unicodedata.category` L* AND `name` starts "LATIN"),
combining marks (M*, for decomposed tone marks), decimal digits (Nd), whitespace, and
an explicit prosody-punctuation set. Verified live: `a ằ ộ ữ đ Đ 1` all KEEP,
`中 文 ★` all drop. So Vietnamese precomposed tone marks and đ/Đ survive while stray
CJK and decorative symbols go — the exact split the feature wants.

### Details that matter
- **Prosody punctuation kept** (`. ! ? … , ; : " ' “ ” ‘ ’ « » ( )`) is a superset of
  what `tts/base.py::_SENTENCE_RE` keys on, so cleaning can't break sentence chunking.
  A test runs the cleaner output back through `split_sentences` to prove it.
- **`\n\n` preserved** (paragraph structure `split_sentences` needs); runs of spaces
  collapsed, blank runs capped at one blank line.
- **Dropped visible symbol → space** (so "A★B" → "A B", not "AB"); **control/format/
  zero-width → removed entirely** (so a ZWSP inside a word vanishes without splitting
  it — tested with a real U+200B).
- **Dashes** (— – ― − -) normalised to a plain hyphen so dialogue dashes survive.

### Tests: 17, pure
The load-bearing one: a real Vietnamese paragraph (tone marks, đ, digits, dialogue,
`1234`) returns **byte-identical**. Plus idempotence, emoji/decorative/CJK/markdown/
zero-width removal, dash normalisation, whitespace tidy, symbol-only → empty, and the
downstream-chunking-survives check.

Verified: `385 passed, 5 deselected` (was 368); `ruff check` clean.

### Reconciliation with the plan (which landed after the cleaner was built)
The plan (`017.01-INITIAL-PLAN.md`) matches this implementation — same module, same
`_keep`/`_is_latin` predicate, same chokepoint decision (Step 2, `clean: bool=True`
inside `synthesize_chapter`). Two deltas:
- **NFC normalization** — the plan recommends a `unicodedata.normalize("NFC", text)`
  Step 0 (composes stray combining marks), flagged explicitly as *the user's call*.
  This build does NOT include it: the `M*`-keep branch already preserves decomposed
  tone marks, and real translator output is NFC so it's a practical no-op. Surfaced to
  the user; trivial to add if wanted.
- **Full-width digits** — the plan drops non-ASCII digits (`Nd and isascii()`); this
  build keeps all `Nd`. Cosmetic — full-width digits are ~absent from VN novel text
  and "a digit is a digit" is defensible either way. Left as-is.

## [STEP 2] synthesize_chapter hook — 2026-07-18
Plan 017.01 Step 2. Edited `src/noveltrans/tts/base.py`; tests in `tests/test_tts.py`.

- Added `clean: bool = True` to `synthesize_chapter`, applied to the joined
  `f"{title}\n\n{body}"` **immediately before `split_sentences`** — the single
  chokepoint, so both `AudioWorker` paths and the CLI get it for free.
- Ordering matters: cleaning the exact string `split_sentences` consumes (not title/body
  separately) keeps the kept-punctuation set and the chunker from drifting apart.
- Default-on with an escape hatch (`clean=False`); non-destructive (only the local
  `text` is cleaned).

### Tests: 3 new (42 in test_tts.py, 388 total)
Using the existing `FakeTtsEngine`, which records each chunk it's handed:
- `clean=True` (default) — engine never sees `★ 😀 中`; Vietnamese survives.
- `clean=False` — text passes through untouched (the escape hatch).
- **symbols-only chapter cleans to `""`** → hits the existing
  `TtsError("… không có nội dung …")` gracefully, no crash on an empty concat. This was
  the edge case the plan flagged; confirmed it degrades to the existing error path.

Verified: `388 passed, 5 deselected` (was 385); `ruff check` clean.
