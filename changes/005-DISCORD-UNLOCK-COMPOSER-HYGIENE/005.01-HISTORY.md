# 005 — Discord unlock composer hygiene — History

## 2026-07-14 — Clear stale draft + hold window after send

**Context:** Follow-up to 004 (Discord auto-unlock). Two issues the user raised
after live use:

1. Discord persists a **per-channel draft**. A fragment left in the composer from a
   previous run would get the new `/mochuong` command appended to it, colliding old
   and new input.
2. Closing the browser immediately after send could drop the interaction before the
   bot processed it.

**Changes (`discord_unlock.py`, `run_unlock`):**
- After clicking the composer, `_clear_composer(page)` wipes any leftover draft
  (select-all + backspace) before typing the new command — every run starts empty.
- Added `_POST_SEND_WAIT_MS = 12_000`: once the command is confirmed sent (composer
  emptied), hold the window open ~12s so the bot can receive/process the interaction
  before teardown. Sits inside the user's requested 10–15s.

**Interaction with 004's resume race fix:** the 12s in-browser hold now precedes the
`unlocked` signal, so by resume time the bot has already had ~12s; the 6s
`_UNLOCK_SETTLE_MS` before resume is now partly redundant but kept as cheap insurance
against the double-unlock.

**Verification:** ruff clean, 180 tests pass. Still not exercised against a real
cap-code (same open item as 004) — the draft-clear and hold are simple enough to be
confident in at dry-run level.
